4.1 Suivi de l'Activité du Serveur

4.1.1 Utilisation des Vues Dynamiques

SQL Server fournit des vues dynamiques qui permettent de surveiller divers aspects de l'activité du serveur. Voici quelques-unes des vues dynamiques les plus couramment utilisées :

  • sys.dm_exec_sessions : Cette vue donne des informations sur chaque session active sur le serveur, telles que l'ID de session, l'utilisateur connecté, l'état de la session, etc.

  • sys.dm_exec_requests : Permet de voir les requêtes en cours d'exécution sur le serveur, avec des détails tels que l'ID de session, l'état de la requête, le texte de la requête, etc.

  • sys.dm_os_performance_counters : Donne accès à divers compteurs de performance SQL Server, comme le nombre de connexions, le taux de transfert de données, etc.

  • sys.dm_db_index_usage_stats : Fournit des informations sur l'utilisation des index, y compris le nombre de lectures, d'écritures et de scans effectués sur chaque index.

4.1.2 Configuration de la Journalisation

La journalisation dans SQL Server peut être configurée pour enregistrer différentes actions telles que les connexions réussies/échouées, les modifications de schéma, les erreurs système, etc. Voici comment configurer la journalisation :

  1. Ouvrir SSMS : Lancez SQL Server Management Studio.

  2. Connectez-vous au serveur : Connectez-vous au serveur SQL Server où vous souhaitez configurer la journalisation.

  3. Sélectionnez la base de données : Choisissez la base de données pour laquelle vous souhaitez configurer la journalisation.

  4. Configuration de la journalisation : Cliquez avec le bouton droit sur la base de données sélectionnée, sélectionnez "Propriétés", puis accédez à l'onglet "Options".

  5. Activez la journalisation : Dans la section "Journalisation", activez les options de journalisation selon vos besoins. Vous pouvez choisir d'activer la journalisation pour les connexions, les modifications de schéma, les erreurs système, etc.

  6. Définissez les options de journalisation avancées : Selon votre configuration, vous pouvez également définir des options avancées telles que la rétention des journaux, l'emplacement des fichiers journaux, etc.

  7. Appliquez les modifications : Une fois les options configurées, cliquez sur "OK" pour appliquer les modifications.

Une fois la journalisation configurée, SQL Server enregistrera les actions spécifiées dans les journaux système appropriés, ce qui vous permettra de surveiller l'activité du serveur et de diagnostiquer les problèmes éventuels.

4.2 Gestion des Erreurs et des Alertes

4.2.1 Configuration des Alertes

Les alertes dans SQL Server sont utilisées pour être averti lorsqu'un événement spécifique se produit. Voici comment configurer des alertes :

  1. Ouvrir SSMS : Lancez SQL Server Management Studio.

  2. Connectez-vous au serveur : Connectez-vous au serveur SQL Server où vous souhaitez configurer des alertes.

  3. Navigation dans l'Explorateur d'Objets : Développez le nœud "Gestion" dans l'Explorateur d'Objets, puis le nœud "Alertes".

  4. Créer une nouvelle alerte : Cliquez avec le bouton droit sur "Alertes" et sélectionnez "Nouvelle Alerte".

  5. Configurer l'alerte : Dans la fenêtre de configuration de l'alerte, donnez-lui un nom descriptif et choisissez le type de condition déclencheur, comme une erreur de niveau de gravité spécifique, un avertissement de performance, etc.

  6. Définissez les paramètres de déclenchement : Configurez les paramètres spécifiques de déclenchement pour l'alerte, tels que le seuil de performance ou le numéro d'erreur.

  7. Définissez l'action à entreprendre : Spécifiez l'action à entreprendre lorsque l'alerte est déclenchée, comme envoyer un e-mail, exécuter un travail, etc.

  8. Définissez les options de réponse : Choisissez les options de réponse, telles que la répétition de l'alerte, la durée de l'attente avant de répéter, etc.

  9. Testez l'alerte : Avant de l'enregistrer, vous pouvez tester l'alerte pour vous assurer qu'elle fonctionne comme prévu en cliquant sur le bouton "Test".

  10. Enregistrez l'alerte : Une fois satisfait de la configuration, cliquez sur "OK" pour enregistrer l'alerte.

4.2.2 Traitement des Erreurs

La gestion des erreurs dans SQL Server peut être réalisée à l'aide de blocs TRY...CATCH dans les procédures stockées, ainsi que par la configuration des actions à prendre en cas d'erreur système. Voici comment gérer les erreurs :

  1. Blocs TRY...CATCH : Encadrez le code potentiellement générateur d'erreurs dans un bloc TRY. Si une erreur se produit, elle est capturée par le bloc CATCH, où vous pouvez effectuer des actions telles que l'enregistrement dans un journal, l'envoi d'e-mails d'alerte, etc.

    Exemple :

    sql
    BEGIN TRY
    -- Code susceptible de générer des erreurs
END TRY
BEGIN CATCH
    -- Gestion des erreurs
END CATCH

  2. Configuration des actions en cas d'erreur système : Dans les propriétés du serveur dans SSMS, vous pouvez configurer des actions spécifiques à prendre en cas d'erreur système, telles que l'envoi d'un e-mail, l'exécution d'une procédure stockée, etc.

    • Ouvrez SSMS.
    • Connectez-vous au serveur SQL Server.
    • Cliquez avec le bouton droit sur le serveur dans l'Explorateur d'Objets et sélectionnez "Propriétés".
    • Accédez à la section "Configuration des erreurs".
    • Configurez les actions à prendre en cas d'erreur système.

En combinant la configuration des alertes et la gestion des erreurs dans votre code, vous pouvez mettre en place un système robuste pour surveiller et réagir aux erreurs dans SQL Server.

4.3 Analyse des Logs

4.3.1 Interprétation des Logs d'Erreurs

Les logs d'erreurs de SQL Server sont une ressource précieuse pour diagnostiquer les problèmes. Voici comment interpréter les logs d'erreurs dans SSMS :

  1. Ouvrir SSMS : Lancez SQL Server Management Studio.

  2. Connectez-vous au serveur : Connectez-vous au serveur SQL Server dont vous souhaitez analyser les logs d'erreurs.

  3. Navigation dans l'Explorateur d'Objets : Développez le nœud "Gestion" dans l'Explorateur d'Objets, puis le nœud "Journaux".

  4. Afficher les logs d'erreurs : Cliquez avec le bouton droit sur "Logs SQL Server" et sélectionnez "Vue des journaux".

  5. Analysez les entrées : Les entrées dans les logs d'erreurs fournissent des informations sur les événements ayant eu lieu sur le serveur, tels que les erreurs de connexion, les échecs de requêtes, etc. Recherchez les messages d'erreur spécifiques et les codes d'état pour identifier les problèmes.

  6. Utilisez les filtres : SSMS vous permet de filtrer les logs d'erreurs par date, niveau de gravité, mot-clé, etc., ce qui peut vous aider à trouver plus facilement les entrées pertinentes.

  7. Corrigez les problèmes : Une fois les problèmes identifiés, prenez les mesures nécessaires pour les corriger, qu'il s'agisse de modifier des requêtes, de réparer des connexions, etc.

4.3.2 Détection des Problèmes de Performance

Les logs SQL Server peuvent également être utilisés pour détecter les problèmes de performance. Voici quelques étapes pour cela :

  1. Examiner les entrées de performance : En plus des erreurs, les logs SQL Server enregistrent également des informations sur les performances, telles que les temps d'exécution des requêtes, les verrous, etc. Analysez ces entrées pour repérer les goulots d'étranglement et les requêtes lentes.

  2. Utiliser les outils de diagnostic : SSMS propose des outils de diagnostic intégrés, tels que l'Assistant d'Analyse de la Base de Données, qui peuvent vous aider à identifier les problèmes de performance en analysant les logs et en fournissant des recommandations.

  3. Surveiller les métriques de performance : En plus des logs, surveillez les métriques de performance en temps réel à l'aide des vues dynamiques et des compteurs de performance SQL Server. Cela vous permettra de détecter les problèmes dès qu'ils surviennent.

  4. Optimiser les requêtes : Une fois les problèmes de performance identifiés, optimisez les requêtes lentes en réécrivant le code, en ajoutant des index, en ajustant les paramètres de configuration, etc.

En analysant régulièrement les logs SQL Server et en surveillant les performances du serveur, vous pouvez identifier et résoudre rapidement les problèmes, ce qui contribue à maintenir une base de données SQL Server performante et fiable.
