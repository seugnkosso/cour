Voici un tutoriel complet et approfondi sur la configuration initiale de Microsoft SQL Server, en se concentrant sur la configuration des paramètres de serveur :

A. Configuration des paramètres de serveur :

  1. Mémoire allouée :

    • Ouvrez SQL Server Management Studio (SSMS).
    • Connectez-vous à votre instance SQL Server.
    • Cliquez avec le bouton droit sur le serveur dans l'Explorateur d'objets et sélectionnez "Propriétés".
    • Dans la fenêtre "Propriétés du serveur", sélectionnez "Mémoire" dans le volet de gauche.
    • Définissez les valeurs maximale et minimale de la mémoire du serveur selon les besoins de votre système.
    • Assurez-vous de laisser suffisamment de mémoire pour le système d'exploitation et d'autres applications s'exécutant sur le serveur.

  2. Taille des fichiers de données et de transaction :

    • Pour définir la taille des fichiers de données, utilisez la clause SIZE dans vos commandes CREATE DATABASE ou ALTER DATABASE.
    • Pour définir la taille des fichiers de transaction, utilisez la clause SIZE dans la définition de vos fichiers de journal ou ajustez la taille de votre fichier journal en utilisant SSMS.
    • Pour ajuster la taille des fichiers existants, utilisez la commande ALTER DATABASE.

  3. Options de démarrage :

    • Les options de démarrage sont configurées lors de l'installation de SQL Server mais peuvent également être modifiées par la suite.
    • Pour modifier les options de démarrage, ouvrez la configuration SQL Server à partir du menu Démarrer.
    • Dans l'onglet "Services", sélectionnez votre instance de SQL Server, puis cliquez sur "Propriétés".
    • Vous pouvez modifier les options de démarrage telles que le mode de démarrage (manuel, automatique, désactivé), le compte de démarrage, etc.
    • Assurez-vous de comprendre les implications de chaque option de démarrage avant d'apporter des modifications.

Assurez-vous de prendre des sauvegardes avant de modifier des paramètres de configuration critiques, et testez toujours les modifications dans un environnement de test avant de les appliquer en production.

B. Configuration des comptes de connexion :

  1. Création de comptes de connexion :

    • Ouvrez SQL Server Management Studio (SSMS).
    • Connectez-vous à votre instance SQL Server en utilisant un compte avec des privilèges suffisants pour créer de nouveaux comptes de connexion.
    • Dans l'Explorateur d'objets, développez le dossier "Sécurité" puis le sous-dossier "Connexions".
    • Cliquez avec le bouton droit sur "Connexions" et sélectionnez "Nouvelle connexion".
    • Dans la fenêtre "Nouvelle connexion", spécifiez le nom du compte de connexion, choisissez s'il s'agit d'un compte Windows ou SQL Server, et définissez un mot de passe si nécessaire.
    • Configurez les autres paramètres selon les besoins, comme la stratégie de mot de passe, les options de connexion, etc.
    • Cliquez sur "OK" pour créer le compte de connexion.

  2. Attribution de rôles et de permissions :

    • Une fois le compte de connexion créé, vous pouvez lui attribuer des rôles et des permissions appropriés.
    • Pour cela, dans SSMS, accédez au dossier "Sécurité" puis "Connexions", trouvez le compte de connexion que vous venez de créer, cliquez avec le bouton droit dessus et sélectionnez "Propriétés".
    • Dans la fenêtre des propriétés du compte de connexion, accédez à l'onglet "Appartenances aux rôles de serveur" pour lui attribuer des rôles de serveur, tels que sysadmin, dbcreator, etc.
    • Pour définir des permissions au niveau de la base de données, accédez à l'onglet "Mappages de l'utilisateur" et sélectionnez la ou les bases de données auxquelles vous souhaitez que le compte de connexion accède.
    • Dans l'onglet "Rôles de base de données", attribuez les rôles de base de données appropriés, tels que db_owner, db_datareader, db_datawriter, etc.
    • Configurez également les autorisations spécifiques en fonction des besoins, en utilisant les options disponibles dans les propriétés du compte de connexion.

Veillez à accorder les autorisations avec parcimonie, en suivant le principe du moindre privilège, pour limiter les risques de sécurité et garantir une gestion efficace des accès.

C. Configuration des paramètres de base de données :

  1. Sélection du mode de récupération (Simple, Full, Bulk-Logged) :

    • Le mode de récupération détermine la façon dont les transactions sont journalisées et comment la récupération est effectuée en cas de panne.
    • Pour changer le mode de récupération d'une base de données, utilisez la commande T-SQL ALTER DATABASE suivie de l'option SET RECOVERY.
    • Par exemple, pour passer à un mode de récupération complet :
      sql
      ALTER DATABASE [NomDeLaBaseDeDonnées] SET RECOVERY FULL;
    • Assurez-vous de comprendre les implications de chaque mode de récupération en termes de performances, de capacité de récupération et de gestion des journaux de transaction.

  2. Configuration des options de compatibilité :

    • Les options de compatibilité déterminent le comportement de la base de données en termes de syntaxe et de comportement des fonctionnalités.
    • Pour modifier l'option de compatibilité d'une base de données, utilisez la commande T-SQL ALTER DATABASE suivie de l'option COMPATIBILITY_LEVEL.
    • Par exemple, pour définir le niveau de compatibilité sur SQL Server 2019 (compatibilité niveau 150) :
      sql
      ALTER DATABASE [NomDeLaBaseDeDonnées] SET COMPATIBILITY_LEVEL = 150;
    • Assurez-vous de comprendre les changements de comportement associés à chaque niveau de compatibilité et de les tester soigneusement.

  3. Activation des options avancées (ex. Compression, Partitioning) :

    • SQL Server propose diverses options avancées telles que la compression des données, le partitionnement de table, l'indexation avancée, etc., qui peuvent améliorer les performances et l'efficacité de la base de données.
    • Pour activer la compression des données, utilisez la clause ROW ou PAGE dans la définition de vos index ou tables.
    • Pour le partitionnement de table, créez des fonctions de partitionnement et des schémas de partitionnement, puis utilisez-les dans la définition de votre table.
    • Assurez-vous de comprendre les exigences et les limitations de chaque option avancée, ainsi que leur impact sur les performances et la maintenance.

Veillez à documenter soigneusement les changements de configuration et à effectuer des tests approfondis avant de les appliquer en production.

D. Configuration de la sauvegarde et de la restauration :

  1. Planification des sauvegardes régulières :

    • Établissez un plan de sauvegarde qui répond aux besoins de votre organisation en termes de récupération et de protection des données.
    • Identifiez les bases de données à sauvegarder, la fréquence des sauvegardes (quotidiennes, hebdomadaires, etc.) et le type de sauvegarde à effectuer (complète, différentielle, transactionnelle).
    • Utilisez SQL Server Management Studio (SSMS) ou des scripts T-SQL pour créer des plans de sauvegarde automatisés en utilisant des travaux SQL Server Agent.
    • Assurez-vous de stocker les sauvegardes dans des emplacements sécurisés et redondants pour éviter la perte de données.

  2. Configuration des stratégies de rétention :

    • Définissez des stratégies de rétention pour gérer l'espace de stockage des sauvegardes et maintenir un historique approprié des sauvegardes.
    • Utilisez des plans de maintenance ou des scripts T-SQL pour nettoyer les sauvegardes obsolètes en fonction de la politique de rétention définie.
    • Assurez-vous de conserver suffisamment de sauvegardes pour répondre aux exigences de récupération de votre organisation tout en optimisant l'utilisation de l'espace de stockage.

  3. Test de la restauration de sauvegardes :

    • Il est essentiel de tester régulièrement la restauration des sauvegardes pour garantir leur intégrité et leur efficacité en cas de panne.
    • Créez un environnement de test séparé où vous pouvez restaurer les sauvegardes et valider l'intégrité des données.
    • Utilisez des scripts T-SQL ou SSMS pour restaurer les sauvegardes dans votre environnement de test, puis effectuez des tests de validation pour vous assurer que les données sont restaurées correctement et que les applications fonctionnent comme prévu.
    • Documentez et automatisez le processus de test de restauration pour qu'il soit régulièrement exécuté et suivi.

En suivant ces étapes, vous pouvez mettre en place une stratégie de sauvegarde et de restauration robuste qui protège les données de votre organisation et minimise les temps d'arrêt en cas de sinistre.
