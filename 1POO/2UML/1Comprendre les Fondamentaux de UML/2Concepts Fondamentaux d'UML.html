Définition de classe

Notions fondamentales :

  • Classe : Une classe est un modèle ou un plan pour créer des objets. Elle définit les propriétés (attributs) et les comportements (méthodes) des objets qui seront créés à partir d'elle.

  • Objet : Un objet est une instance d'une classe. Il est créé en utilisant le modèle fourni par la classe et possède des valeurs spécifiques pour ses attributs.

Structure d'une classe

Une classe UML est représentée par un rectangle divisé en trois compartiments :

  1. Nom de la classe : En haut du rectangle, on place le nom de la classe.

    plaintext
    +------------------+
|   NomDeLaClasse  |
+------------------+
|  Attributs       |
|  --------------  |
|  Méthodes        |
+------------------+

  2. Attributs : Ce compartiment contient la liste des attributs de la classe. Un attribut est une propriété de la classe. Chaque attribut a un nom et un type.

    plaintext
    +------------------+
|   NomDeLaClasse  |
+------------------+
|  - attribut1: type1 |
|  - attribut2: type2 |
|  ...             |
|  - attributN: typeN |
|  --------------  |
|  Méthodes        |
+------------------+

  3. Méthodes : Ce compartiment contient la liste des méthodes de la classe. Une méthode est un comportement de la classe. Chaque méthode a un nom, des paramètres (le cas échéant), un type de retour et une implémentation.

    plaintext
    +------------------+
|   NomDeLaClasse  |
+------------------+
|  - attribut1: type1 |
|  - attribut2: type2 |
|  ...             |
|  - attributN: typeN |
|  --------------  |
|  + méthode1(param1: type1): retour1 |
|  + méthode2(param2: type2): retour2 |
|  ...             |
|  + méthodeN(paramN: typeN): retourN |
+------------------+

Exemples pratiques

Exemple 1 : Définition d'une classe "Personne"

plaintext
+------------------+
|     Personne     |
+------------------+
|  - nom: String    |
|  - age: int       |
|  - adresse: String|
|  --------------  |
|  + sePresenter(): void |
|  + changerAdresse(nouvelleAdresse: String): void |
+------------------+

Dans cet exemple, la classe "Personne" a trois attributs (nom, age, adresse) et deux méthodes (sePresenter, changerAdresse).

Exemple 2 : Définition d'une classe "Voiture"

plaintext
+------------------+
|      Voiture     |
+------------------+
|  - modele: String |
|  - annee: int     |
|  - couleur: String|
|  --------------  |
|  + demarrer(): void   |
|  + accelerer(vitesse: int): void |
|  + arreter(): void |
+------------------+

Dans cet exemple, la classe "Voiture" a trois attributs (modele, annee, couleur) et trois méthodes (demarrer, accelerer, arreter).

Ces exemples illustrent comment définir une classe en UML avec ses attributs et méthodes. Vous pouvez adapter ces modèles de classe en fonction des besoins spécifiques de votre application.

Instance de classe

Une instance de classe est un objet créé à partir d'une classe. Chaque objet est une occurrence spécifique de la classe, possédant des valeurs particulières pour ses attributs. Voici comment vous pouvez représenter cela en UML.

Représentation UML :

plaintext
+------------------+
|   NomDeLaClasse  |
+------------------+
|  - attribut1: type1 |
|  - attribut2: type2 |
|  ...             |
|  - attributN: typeN |
|  --------------  |
|  + méthode1(param1: type1): retour1 |
|  + méthode2(param2: type2): retour2 |
|  ...             |
|  + méthodeN(paramN: typeN): retourN |
+------------------+
|     objet1       |
+------------------+
|     objet2       |
+------------------+
|     ...          |
+------------------+

Chaque objet (objet1, objet2, etc.) est une instance de la classe "NomDeLaClasse" avec des valeurs spécifiques pour ses attributs.

Création et utilisation d'objets

Pour créer un objet en UML, on utilise une flèche avec une étiquette de stéréotype « instanciation ». Cette flèche pointe de la classe vers l'objet nouvellement créé.

Exemple UML :

plaintext
+------------------+
|   Personne       |
+------------------+
|  - nom: String    |
|  - age: int       |
|  - adresse: String|
|  --------------  |
|  + sePresenter(): void |
|  + changerAdresse(nouvelleAdresse: String): void |
+------------------+
          |
          | instanciation
          v
+------------------+
|   personne1      |
+------------------+

Dans cet exemple, "personne1" est une instance de la classe "Personne". L'instanciation est représentée par une flèche pointant de la classe "Personne" vers l'objet "personne1".

Caractéristiques des objets

Les objets ont plusieurs caractéristiques importantes en UML :

  1. Identité : Chaque objet a une identité unique. Deux objets peuvent avoir les mêmes valeurs pour leurs attributs, mais s'ils ont des identités différentes, ce sont deux objets distincts.

  2. Attributs : Les objets ont des valeurs spécifiques pour leurs attributs, déterminées lors de leur création. Ces valeurs peuvent changer au fil du temps.

  3. Méthodes : Les objets peuvent exécuter les méthodes définies dans leur classe. Chaque objet utilise le modèle de comportement fourni par sa classe.

  4. Encapsulation : Les objets encapsulent leurs états (attributs) et comportements (méthodes) de manière à ce qu'ils soient accessibles uniquement à travers des interfaces bien définies.

  5. Relation avec d'autres objets : Les objets peuvent interagir entre eux en utilisant des méthodes et en échangeant des messages. Ces relations peuvent être représentées en UML à l'aide de diagrammes de séquence, de collaboration, etc.

En résumé, la création et l'utilisation d'objets en UML impliquent l'instanciation d'une classe pour former des instances uniques, chaque instance ayant des valeurs spécifiques pour ses attributs. Les objets interagissent les uns avec les autres pour réaliser les fonctionnalités définies dans leurs classes respectives.

Attributs de classe

Les attributs de classe sont des propriétés partagées par toutes les instances d'une classe. Ils sont généralement définis avec le mot-clé "static". Les attributs de classe appartiennent à la classe elle-même plutôt qu'à une instance spécifique.

Représentation UML :

plaintext
+------------------+
|   NomDeLaClasse  |
+------------------+
|  + attributClasse: type |
|  --------------  |
|  + methodeClasse(): retour |
+------------------+

Attributs d'instance

Les attributs d'instance sont des propriétés spécifiques à chaque instance d'une classe. Ils définissent l'état de l'objet. Chaque instance a ses propres valeurs d'attributs.

Représentation UML :

plaintext
+------------------+
|   NomDeLaClasse  |
+------------------+
|  - attributInstance: type |
|  --------------  |
|  + methodeInstance(): retour |
+------------------+

Méthodes de classe

Les méthodes de classe sont des comportements partagés par toutes les instances d'une classe. Elles sont également définies avec le mot-clé "static". Ces méthodes peuvent accéder uniquement aux attributs de classe, pas aux attributs d'instance.

Représentation UML :

plaintext
+------------------+
|   NomDeLaClasse  |
+------------------+
|  + attributClasse: type |
|  --------------  |
|  + methodeClasse(): retour |
|  + methode2Classe(param: type): retour |
+------------------+

Méthodes d'instance

Les méthodes d'instance sont des comportements spécifiques à chaque instance d'une classe. Elles peuvent accéder et manipuler les attributs d'instance de l'objet sur lequel elles sont appelées.

Représentation UML :

plaintext
+------------------+
|   NomDeLaClasse  |
+------------------+
|  - attributInstance: type |
|  --------------  |
|  + methodeInstance(): retour |
|  + methode2Instance(param: type): retour |
+------------------+

Exemples pratiques

Exemple 1 : Attributs et Méthodes de classe et d'instance

plaintext
+------------------+
|   CompteBancaire  |
+------------------+
|  + tauxInteret: double |
|  - solde: double  |
|  --------------  |
|  + appliquerInteret(): void |
|  - debiter(montant: double): void |
|  + crediter(montant: double): void |
+------------------+

Dans cet exemple, "tauxInteret" est un attribut de classe, partagé par toutes les instances de la classe. "solde" est un attribut d'instance, spécifique à chaque compte bancaire. Les méthodes "appliquerInteret", "debiter" et "crediter" sont respectivement des méthodes de classe, pouvant accéder à "tauxInteret", et des méthodes d'instance, pouvant accéder et manipuler "solde".

Exemple 2 : Utilisation d'attributs et de méthodes

plaintext
CompteBancaire tauxInteret = 0.05; // attribut de classe
CompteBancaire compteA = new CompteBancaire(1000); // attribut d'instance
CompteBancaire compteB = new CompteBancaire(2000); // attribut d'instance

compteA.crediter(500); // méthode d'instance
compteA.appliquerInteret(); // méthode de classe

compteB.debiter(100); // méthode d'instance

Dans cet exemple, "tauxInteret" est un attribut de classe partagé par toutes les instances. Les objets "compteA" et "compteB" ont des valeurs distinctes pour leur attribut d'instance "solde". Les méthodes sont appelées sur des instances spécifiques pour manipuler leurs états respectifs.

Relation de base

L'association en UML représente une relation sémantique entre deux classes. Elle peut être simple ou complexe, indiquant comment les objets de ces classes sont liés.

Représentation UML :

plaintext
+-----------------+      +-----------------+
|   ClasseA       |      |   ClasseB       |
+-----------------+      +-----------------+
|  - attributA: type |    |  - attributB: type |
|  --------------   |    |  --------------   |
|  + methodeA(): retour |  |  + methodeB(): retour |
+-----------------+      +-----------------+
         |
         | Association
         |
         v
+-----------------------+
|      Association     |
|  ------------------   |
|  - attributAssoc: type |
|  + methodeAssoc(): retour |
+-----------------------+

Dans cet exemple, "ClasseA" et "ClasseB" sont liées par une association, représentée par la flèche pointant de l'une à l'autre. L'association peut également avoir ses propres attributs et méthodes, distincts de ceux des classes associées.

Multiplicité des associations

La multiplicité spécifie le nombre d'objets d'une classe associée à un objet de l'autre classe. Elle est indiquée avec des nombres ou des intervalles.

Représentation UML avec Multiplicité :

plaintext
+-----------------+      +-----------------+
|   ClasseA       |      |   ClasseB       |
+-----------------+      +-----------------+
|  - attributA: type |    |  - attributB: type |
|  --------------   |    |  --------------   |
|  + methodeA(): retour |  |  + methodeB(): retour |
+-----------------+      +-----------------+
         |
         | Association (1..* signifie "un ou plusieurs")
         |
         v
+-----------------------+
|      Association     |
|  ------------------   |
|  - attributAssoc: type |
|  + methodeAssoc(): retour |
|  - multiplicitéA: 1..* |
|  - multiplicitéB: 0..1 |
+-----------------------+

Dans cet exemple, la multiplicité "1..*" pour "ClasseA" signifie qu'un objet de "ClasseA" est associé à un ou plusieurs objets de "ClasseB". La multiplicité "0..1" pour "ClasseB" signifie qu'un objet de "ClasseB" peut être associé à zéro ou un objet de "ClasseA".

Exemples d'associations

Exemple 1 : Association unidirectionnelle

plaintext
+------------------+       +------------------+
|   Étudiant       |       |   Cours          |
+------------------+       +------------------+
|  - nom: String    |     |  - nomCours: String|
|  - age: int       |     |  - codeCours: String|
|  --------------   |     |  --------------   |
|  + inscrire(cours: Cours): void | |  + enseigner(étudiant: Étudiant): void |
+------------------+       +------------------+

Dans cet exemple, la méthode "inscrire" dans la classe "Étudiant" indique une association unidirectionnelle vers la classe "Cours".

Exemple 2 : Association bidirectionnelle

plaintext
+------------------+       +------------------+
|   Étudiant       |       |   Cours          |
+------------------+       +------------------+
|  - nom: String    |     |  - nomCours: String|
|  - age: int       |     |  - codeCours: String|
|  --------------   |     |  --------------   |
|  + inscrire(cours: Cours): void | |  + enseigner(étudiant: Étudiant): void |
|  + getCoursInscrits(): List<Cours> | |  + getÉtudiants(): List<Étudiant> |
+------------------+       +------------------+

Dans cet exemple, la méthode "inscrire" dans la classe "Étudiant" et la méthode "enseigner" dans la classe "Cours" indiquent une association bidirectionnelle. Les méthodes "getCoursInscrits" et "getÉtudiants" permettent d'accéder aux objets associés des deux côtés de l'association.

Ces exemples illustrent comment représenter et utiliser des associations en UML, en montrant les relations entre les classes et comment les objets peuvent interagir à travers ces associations.

Agrégation vs Association

L'agrégation est une forme spécifique d'association en UML, indiquant qu'une classe est composée d'autres classes. L'agrégation est une relation de type « tout-pouvant-être-partie-de ». Elle est souvent représentée par un losange creux reliant la classe agrégeante à la classe agrégée.

Représentation UML de l'Association :

plaintext
+------------------+       +------------------+
|   ClasseA        |       |   ClasseB        |
+------------------+       +------------------+
|  - attributA: type |     |  - attributB: type|
|  --------------   |     |  --------------   |
|  + methodeA(): retour | |  + methodeB(): retour |
+------------------+       +------------------+
         |
         | Association (lien simple)
         |
         v
+--------------------------+
|      Association        |
|  ---------------------  |
|  - attributAssoc: type  |
|  + methodeAssoc(): retour|
+--------------------------+

Représentation UML de l'Agrégation :

plaintext
+------------------+       +------------------+
|   ClasseA        |       |   ClasseB        |
+------------------+       +------------------+
|  - attributA: type |     |  - attributB: type|
|  --------------   |     |  --------------   |
|  + methodeA(): retour | |  + methodeB(): retour |
|  + objetsB: List<ClasseB> | <-- Agrégation
+------------------+

La principale différence réside dans le sens de la relation. Dans une association simple, les objets peuvent exister indépendamment les uns des autres. Dans une agrégation, la classe agrégante peut contenir ou posséder des objets de la classe agrégée, mais ces objets peuvent exister indépendamment.

Exemples concrets

Exemple 1 : Agrégation "École" et "Étudiant"

plaintext
+------------------+       +------------------+
|      École       |       |     Étudiant     |
+------------------+       +------------------+
|  - nom: String    |     |  - nom: String    |
|  - adresse: String|     |  - age: int       |
|  + getÉtudiants(): List<Étudiant> | |  --------------   |
+------------------+       |  + inscrire(): void|
                          +------------------+

Dans cet exemple, l'agrégation est représentée par la méthode "getÉtudiants" dans la classe "École", qui retourne une liste d'objets "Étudiant". Cela indique que des étudiants peuvent être associés à une école, mais ils peuvent également exister indépendamment de l'école.

Exemple 2 : Agrégation "Entreprise" et "Employé"

plaintext
+------------------+       +------------------+
|    Entreprise    |       |      Employé     |
+------------------+       +------------------+
|  - nom: String    |     |  - nom: String    |
|  - adresse: String|     |  - salaire: double|
|  + getEmployés(): List<Employé> | |  --------------   |
+------------------+       |  + travailler(): void|
                          +------------------+

Dans cet exemple, l'agrégation est représentée par la méthode "getEmployés" dans la classe "Entreprise", qui retourne une liste d'objets "Employé". Cela indique que des employés peuvent être associés à une entreprise, mais ils peuvent également travailler indépendamment de l'entreprise.

Ces exemples montrent comment utiliser l'agrégation pour modéliser des relations de « tout-pouvant-être-partie-de » entre des classes en UML. L'agrégation est utile pour représenter des structures de composition où une classe peut contenir d'autres classes, mais où ces classes peuvent également exister indépendamment.

Composition vs Agrégation

La composition est une autre forme de relation en UML, similaire à l'agrégation, mais avec une sémantique plus forte. Elle indique qu'une classe est composée d'autres classes, et que les objets de la classe composée ne peuvent pas exister indépendamment de la classe composite. En d'autres termes, la composition implique une relation « tout-faisant-partie-de » avec une forte dépendance.

Représentation UML de l'Agrégation :

plaintext
+------------------+       +------------------+
|   ClasseA        |       |   ClasseB        |
+------------------+       +------------------+
|  - attributA: type |     |  - attributB: type|
|  --------------   |     |  --------------   |
|  + methodeA(): retour | |  + methodeB(): retour |
|  + objetsB: List<ClasseB> | <-- Agrégation
+------------------+

Représentation UML de la Composition :

plaintext
+------------------+       +------------------+
|   ClasseA        |       |   ClasseB        |
+------------------+       +------------------+
|  - attributA: type |     |  - attributB: type|
|  --------------   |     |  --------------   |
|  + methodeA(): retour | |  + methodeB(): retour |
|  + objetB: ClasseB   | <-- Composition
+------------------+

La principale distinction entre composition et agrégation réside dans le degré de dépendance entre la classe composite et la classe composée. Avec la composition, la classe composée ne peut pas exister sans la classe composite, tandis qu'avec l'agrégation, la classe composée peut exister indépendamment.

Cas d'utilisation de la Composition

Exemple : Composition "Ordinateur" et "Processeur"

plaintext
+------------------+       +------------------+
|    Ordinateur    |       |    Processeur    |
+------------------+       +------------------+
|  - marque: String |     |  - modèle: String |
|  - ram: int       |     |  - vitesse: double|
|  + allumer(): void | |  + calculer(): void |
|  + processeur: Processeur | <-- Composition
+------------------+

Dans cet exemple, la classe "Ordinateur" est composée d'un objet de la classe "Processeur". La composition est indiquée par l'attribut "processeur" dans la classe "Ordinateur". Cela signifie que l'objet "Processeur" ne peut pas exister indépendamment d'un objet "Ordinateur". Si l'ordinateur est détruit, le processeur associé est également détruit.

Avantages de la Composition

  1. Encapsulation renforcée : La composition renforce l'encapsulation en garantissant qu'un objet composé ne peut pas être utilisé en dehors de son contexte.

  2. Gestion des cycles de vie : La composition simplifie la gestion des cycles de vie des objets. Lorsqu'un objet composite est détruit, les objets composés sont également détruits.

  3. Clarté sémantique : En utilisant la composition, vous indiquez clairement que les objets composés sont une partie intégrante de l'objet composite.

Conclusion

La composition en UML est une relation forte qui indique qu'une classe est composée d'autres classes, et que les objets de la classe composée dépendent de la classe composite. Elle offre une encapsulation renforcée et une gestion simplifiée des cycles de vie des objets. Utilisez la composition lorsque vous souhaitez indiquer une forte dépendance entre les objets composés et l'objet composite.

Principes de l'héritage

L'héritage en UML permet de créer une nouvelle classe basée sur une classe existante, appelée classe parente ou classe de base. La nouvelle classe, appelée classe dérivée ou classe fille, hérite des propriétés et des comportements de la classe parente. L'héritage favorise la réutilisation du code, la modélisation de relations is-a (est-un) et la généralisation/specialisation.

Représentation UML :

plaintext
+------------------+       +------------------+
|  Classe de Base  |       |  Classe Dérivée  |
+------------------+       +------------------+
|  - attributBase: type |   |  - attributDerive: type |
|  --------------   |       |  --------------   |
|  + methodeBase(): retour | |  + methodeDerive(): retour |
+------------------+

Classes abstraites et concrètes

Classes Concrètes

Les classes concrètes sont des classes pour lesquelles il est possible de créer des instances. Elles peuvent contenir des attributs, des méthodes, et peuvent être instanciées.

Représentation UML d'une Classe Concrète :

plaintext
+------------------+
|  ClasseConcrète  |
+------------------+
|  - attribut: type |
|  --------------   |
|  + methode(): retour |
+------------------+

Classes Abstraites

Les classes abstraites sont des classes qui ne peuvent pas être instanciées directement. Elles servent souvent de modèle pour les classes dérivées. Elles peuvent contenir des méthodes abstraites (sans implémentation) qui doivent être implémentées par les classes dérivées.

Représentation UML d'une Classe Abstraite :

plaintext
+------------------+
|  ClasseAbstraite |
+------------------+
|  - attribut: type |
|  --------------   |
|  + methodeAbstraite(): abstract retour |
+------------------+

Utilisation de l'héritage

L'utilisation de l'héritage est appropriée lorsque deux classes partagent une relation is-a (est-un), et que la classe dérivée peut bénéficier de la réutilisation du code de la classe de base.

Exemple d'Héritage :

plaintext
+------------------+       +------------------+
|      Animal      |       |       Chien      |
+------------------+       +------------------+
|  - nom: String    |     |  - race: String   |
|  --------------   |     |  --------------   |
|  + manger(): void | |  + aboyer(): void   |
+------------------+       +------------------+

Dans cet exemple, la classe "Chien" hérite de la classe "Animal". Un chien est un type d'animal, donc la relation est de type is-a. La classe "Chien" hérite des propriétés et des méthodes de la classe "Animal", tout en ajoutant des attributs et des méthodes spécifiques à la classe "Chien".

Limitations de l'héritage

Bien que l'héritage soit une technique puissante, elle présente certaines limitations :

  1. Couplage fort : L'héritage crée un couplage fort entre la classe parente et la classe dérivée. Les modifications dans la classe parente peuvent avoir un impact sur toutes les classes dérivées.

  2. Héritage multiple : Certains langages ne prennent pas en charge l'héritage multiple (quand une classe hérite de plusieurs classes). Cela peut limiter la flexibilité du modèle.

  3. Problèmes de hiérarchies profondes : Des hiérarchies d'héritage profondes peuvent être difficiles à gérer et à comprendre. Une conception incorrecte peut conduire à des structures complexes et difficiles à maintenir.

  4. Héritage pour la réutilisation : L'héritage doit être utilisé avec prudence, et la réutilisation du code peut souvent être mieux gérée par la composition ou d'autres techniques.

En résumé, bien que l'héritage soit un concept puissant en UML, il doit être utilisé judicieusement en fonction des besoins spécifiques de la conception. La compréhension des principes de l'héritage, des classes abstraites et concrètes, ainsi que des limitations, est essentielle pour une utilisation efficace.
