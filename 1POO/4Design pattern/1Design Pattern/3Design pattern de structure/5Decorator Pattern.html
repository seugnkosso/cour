A. Concept de base

1. Extension de fonctionnalités sans altération de la structure

Le Decorator Pattern est un pattern de conception structurel qui permet d'attacher dynamiquement des responsabilités supplémentaires à un objet. Il offre une alternative flexible à l'héritage pour étendre les fonctionnalités d'une classe.

Le principe clé est d'utiliser des classes de décorateurs pour envelopper des composants concrets. Chaque décorateur a une relation avec le composant de base, ce qui permet d'ajouter des fonctionnalités sans altérer la structure.

2. Comparaison avec l'héritage classique

Dans l'héritage classique, l'extension de fonctionnalités se fait par la création de sous-classes. Cependant, cela peut devenir rapidement complexe et entraîner une classe hiérarchie lourde. Le Decorator Pattern offre une alternative plus souple en permettant de combiner différents comportements de manière dynamique.

B. Participants du Decorator Pattern

1. Composant

Le composant est l'interface commune que tous les objets concrets et décorateurs implémentent. Il définit les opérations de base que les décorateurs et les composants concrets doivent mettre en œuvre.

python
interface Composant {
    operation(): void;
}

2. Décorateur

Le décorateur est une classe abstraite ou une interface qui implémente l'interface du composant. Il contient une référence à un objet de type Composant et peut ajouter des responsabilités supplémentaires.

python
abstract class Decorateur implements Composant {
    protected composant: Composant;

    constructor(composant: Composant) {
        this.composant = composant;
    }

    operation(): void {
        this.composant.operation();
    }
}

3. ConcrèteComposant

La classe ConcrèteComposant est une implémentation concrète de l'interface Composant. Elle fournit l'implémentation de base des opérations.

python
class ConcretComposant implements Composant {
    operation(): void {
        // Implémentation de base
    }
}

4. ConcrèteDécorateur

Les classes ConcrèteDécorateur étendent Decorateur et ajoutent des fonctionnalités spécifiques.

python
class ConcretDecorateurA extends Decorateur {
    operation(): void {
        super.operation();
        this.nouvelleFonctionnalite();
    }

    nouvelleFonctionnalite(): void {
        // Nouvelle fonctionnalité
    }
}

Cela permet de combiner plusieurs décorateurs de manière flexible :

python
let composant: Composant = new ConcretComposant();
composant = new ConcretDecorateurA(composant);
composant = new ConcretDecorateurB(composant);
composant.operation();

Ainsi, le Decorator Pattern offre une approche souple et extensible pour ajouter des fonctionnalités à des objets sans altérer leur structure de base.

B. Description des éléments structurels

1. Composant

  • Description : Définit l'interface commune pour les objets concrets et les décorateurs.
  • Méthode :
    • operation(): void : Opération de base que tous les composants (concrets et décorateurs) doivent implémenter.

2. Décorateur

  • Description : Classe abstraite ou interface qui étend le Composant et contient une référence à un objet de type Composant. Ajoute des responsabilités supplémentaires.
  • Attributs :
    • - composant: Composant : La référence à l'objet de type Composant.
  • Méthodes :
    • operation(): void : Implémente l'opération en faisant appel à l'opération du composant.

3. ConcrèteComposant

  • Description : Implémentation concrète de l'interface Composant.
  • Méthodes :
    • operation(): void : Implémentation concrète de l'opération de base.

4. ConcrèteDécorateur

  • Description : Implémente un décorateur concret qui étend le Decorateur.
  • Attributs :
    • - composant: Composant : La référence à l'objet de type Composant.
  • Méthodes :
    • operation(): void : Appelle l'opération du composant et ajoute des fonctionnalités supplémentaires.
    • nouvelleFonctionnalite(): void : Nouvelle fonctionnalité ajoutée par le décorateur.

Ainsi, le Decorator Pattern permet de combiner des composants et des décorateurs de manière flexible tout en maintenant une structure claire et extensible.

A. Étapes générales

1. Création de l'interface Composant

python
# Composant.py
from abc import ABC, abstractmethod

class Composant(ABC):
    @abstractmethod
    def operation(self) -> None:
        pass

2. Implémentation de la classe ConcrèteComposant

python
# ConcretComposant.py
from Composant import Composant

class ConcretComposant(Composant):
    def operation(self) -> None:
        print("Opération de base de ConcretComposant")

3. Création de l'interface Décorateur

python
# Decorateur.py
from Composant import Composant
from abc import ABC, abstractmethod

class Decorateur(Composant, ABC):
    def __init__(self, composant: Composant):
        self._composant = composant

    @abstractmethod
    def operation(self) -> None:
        pass

4. Implémentation de la classe ConcrèteDécorateur

python
# ConcretDecorateur.py
from Decorateur import Decorateur

class ConcretDecorateur(Decorateur):
    def operation(self) -> None:
        super().operation()  # Appel de l'opération du composant
        self.nouvelleFonctionnalite()

    def nouvelleFonctionnalite(self) -> None:
        print("Nouvelle fonctionnalité ajoutée par ConcretDecorateur")

5. Utilisation du Decorator Pattern

python
# main.py
from ConcretComposant import ConcretComposant
from ConcretDecorateur import ConcretDecorateur

# Création d'un composant concret
composant = ConcretComposant()

# Ajout d'une fonctionnalité via un décorateur
composantAvecDecoration = ConcretDecorateur(composant)

# Appel de l'opération sur le composant décoré
composantAvecDecoration.operation()

Dans cet exemple, le Decorator Pattern est utilisé pour ajouter une nouvelle fonctionnalité à un composant sans altérer sa structure. Vous pouvez ajouter d'autres décorateurs pour combiner plusieurs fonctionnalités de manière dynamique. L'utilisation de ce modèle permet une extension flexible des fonctionnalités sans avoir à modifier les classes existantes.

A. Scénario d'utilisation

Supposons que vous développiez un système de café où vous avez différents types de cafés (Composant) et vous souhaitez ajouter des options de décoration telles que le lait, le sucre, etc. sans altérer la structure initiale des cafés.

B. Code source détaillé

1. Interface Composant

python
# Composant.py
from abc import ABC, abstractmethod

class Composant(ABC):
    @abstractmethod
    def cout(self) -> float:
        pass

2. Classe ConcrèteComposant

python
# CafeSimple.py
from Composant import Composant

class CafeSimple(Composant):
    def cout(self) -> float:
        return 2.0  # Prix de base pour un café simple

3. Interface Décorateur

python
# Decorateur.py
from Composant import Composant
from abc import ABC, abstractmethod

class Decorateur(Composant, ABC):
    def __init__(self, composant: Composant):
        self._composant = composant

    @abstractmethod
    def cout(self) -> float:
        pass

4. Classes ConcrèteDécorateur

a. LaitDecorateur

python
# LaitDecorateur.py
from Decorateur import Decorateur

class LaitDecorateur(Decorateur):
    def cout(self) -> float:
        return self._composant.cout() + 1.0  # Coût du lait ajouté

b. SucreDecorateur

python
# SucreDecorateur.py
from Decorateur import Decorateur

class SucreDecorateur(Decorateur):
    def cout(self) -> float:
        return self._composant.cout() + 0.5  # Coût du sucre ajouté

5. Utilisation dans le client

python

from CafeSimple import CafeSimple

# Ajout de sucre

# Calcul du coût final
coutFinal = cafeLaitSucre.cout()
print(f"Coût total du café : {coutFinal}")

Dans cet exemple, vous pouvez créer différents types de cafés (composants) et ajouter des options de décoration (décorateurs) pour calculer le coût final du café avec les options sélectionnées. Le Decorator Pattern permet d'ajouter des fonctionnalités de manière flexible sans modifier la structure des composants de base.

A. Avantages du Decorator Pattern

1. Flexibilité et extensibilité

Le Decorator Pattern offre une flexibilité et une extensibilité considérables dans la conception logicielle. Il permet d'ajouter de nouvelles fonctionnalités à des objets existants sans altérer leur structure de base. Cela facilite l'extension du code sans recourir à une hiérarchie de classes volumineuse.

2. Facilité de combinaison de fonctionnalités

Le pattern permet de combiner différentes fonctionnalités de manière simple et flexible. Vous pouvez ajouter des décorateurs à un composant de base de manière dynamique pour obtenir différentes combinaisons de fonctionnalités. Cela rend le code plus modulaire et facilite la création de configurations variées.

3. Évite la surcharge de classes avec des fonctionnalités spécifiques

Contrairement à l'héritage classique, où l'ajout de fonctionnalités se fait par la création de sous-classes, le Decorator Pattern permet d'éviter la surcharge de classes avec des fonctionnalités spécifiques. Chaque fonctionnalité est encapsulée dans un décorateur distinct, ce qui rend le système plus lisible et maintenable.

VII. Limitations du Decorator Pattern

A. Complexité potentielle

L'utilisation intensive de décorateurs peut rendre le code complexe, surtout lorsque de nombreuses combinaisons de décorateurs sont possibles. La gestion des interactions entre différents décorateurs peut devenir délicate, conduisant à une complexité accrue et à une courbe d'apprentissage plus raide.

B. Risque de sur-décoration

Il est possible de tomber dans le piège de la sur-décoration, où l'ajout excessif de décorateurs peut rendre le système difficile à comprendre et à maintenir. Il est essentiel de maintenir un équilibre et d'utiliser le Decorator Pattern de manière judicieuse pour éviter une complexité inutile.

C. Difficulté à retirer certaines fonctionnalités

Une fois qu'un objet a été décoré, il peut être difficile de retirer certaines fonctionnalités sans altérer la structure existante. La séparation complète de certaines fonctionnalités spécifiques peut nécessiter des ajustements dans la conception.

En résumé, bien que le Decorator Pattern offre de nombreux avantages en termes de flexibilité et d'extensibilité, il est important de l'utiliser avec prudence pour éviter des niveaux excessifs de complexité et de sur-décoration.
