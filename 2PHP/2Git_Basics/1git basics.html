1. Introduction à Git

1.1 Qu'est-ce que Git ?

Git est un système de gestion de version décentralisé conçu pour suivre les changements dans les fichiers source pendant le développement logiciel. Il a été créé par Linus Torvalds en 2005 et est largement utilisé dans le développement de logiciels pour suivre les modifications apportées au code source.

Git fonctionne en enregistrant les modifications de fichiers sous forme de "commits" dans un référentiel. Chaque commit représente un ensemble de changements associés à un point spécifique dans le temps. Les utilisateurs peuvent travailler de manière collaborative sur un projet tout en gardant une trace des modifications, ce qui facilite la gestion des versions du code.

1.2 Avantages de l'utilisation de Git

  • Décentralisation : Chaque utilisateur possède une copie complète de l'historique du projet, permettant un travail hors ligne et une collaboration fluide.

  • Historique complet : Git conserve un historique complet des modifications, facilitant le suivi des changements, la résolution des conflits et le retour en arrière si nécessaire.

  • Branching et merging : Git permet de créer des branches pour développer des fonctionnalités isolément. Ces branches peuvent ensuite être fusionnées de manière propre et efficace.

  • Economie d'espace : Git utilise une structure de données efficace, ne stockant que les différences entre les versions, ce qui économise de l'espace disque.

  • Vitesse : Les opérations de Git sont généralement rapides grâce à son architecture décentralisée et à l'utilisation de branches.

1.3 Différence entre Git et autres systèmes de contrôle de version

Git se distingue des autres systèmes de contrôle de version, tels que SVN ou Mercurial, par sa décentralisation et son approche de gestion des branches.

  • Décentralisation : Contrairement à SVN qui nécessite une connexion constante au serveur, Git permet aux utilisateurs de travailler localement et de synchroniser leurs changements plus tard.

  • Branching : Git facilite la création et la fusion de branches, ce qui simplifie le développement parallèle de fonctionnalités sans perturber le code principal.

  • Efficacité : Git est souvent considéré comme plus rapide que d'autres systèmes, en particulier pour les opérations courantes telles que la création de branches et la fusion.

Ce tutoriel sert d'introduction à Git. Les sections suivantes exploreront plus en détail les commandes et les concepts de Git pour aider les utilisateurs à maîtriser cet outil puissant de gestion de version.

2. Installation et Configuration de Git

2.1 Installation de Git

Sur Linux (Debian/Ubuntu)

bash
sudo apt update
sudo apt install git

Sur macOS

bash
brew install git

Sur Windows

Téléchargez l'installateur depuis Git for Windows et suivez les instructions d'installation.

2.2 Configuration initiale

Une fois Git installé, configurez votre nom d'utilisateur et votre adresse e-mail. Ces informations seront associées à vos commits.

bash
git config --global user.name "Votre Nom"
git config --global user.email "votre@email.com"

2.3 Paramètres de configuration essentiels

Afficher la configuration globale

bash
git config --global --list

Modifier la configuration globale

bash
git config --global core.editor "nom_de_votre_editeur"
git config --global color.ui auto
  • core.editor : Définit l'éditeur de texte par défaut pour les messages de commit.
  • color.ui : Active la coloration syntaxique dans la sortie de la console.

Configuration spécifique à un projet

Vous pouvez également configurer des paramètres spécifiques à un projet en exécutant ces commandes à l'intérieur du répertoire du projet sans l'option --global.

bash
git config user.name "Votre Nom"
git config user.email "votre@email.com"

Ces configurations spécifiques au projet remplacent les configurations globales lorsqu'elles sont présentes.

Cela conclut la section sur l'installation et la configuration initiale de Git. La prochaine partie du tutoriel explorera les concepts de base de Git, tels que l'initialisation de dépôts, la création de commits, et la gestion des branches.

3. Commandes Fondamentales de Git

3.1 git init

Initialise un nouveau dépôt Git dans le répertoire actuel.

bash
git init

3.2 git clone

Clone un dépôt distant existant dans un nouveau répertoire.

bash
git clone <url_du_dépôt>

3.3 git add

Ajoute des modifications de fichiers à l'index pour les inclure dans le prochain commit.

bash
git add <nom_du_fichier>

Pour ajouter tous les fichiers modifiés ou nouveaux, utilisez :

bash
git add .

3.4 git commit

Crée un nouveau commit avec les modifications de l'index.

bash
git commit -m "Message de commit"

3.5 git status

Affiche l'état des fichiers dans le répertoire de travail par rapport au dernier commit.

bash
git status

3.6 git log

Affiche l'historique des commits avec les détails comme l'auteur, la date et le message du commit.

bash
git log

3.7 git diff

Affiche les différences non confirmées entre le répertoire de travail et le dernier commit.

bash
git diff

Pour voir les différences pour un fichier spécifique :

bash
git diff <nom_du_fichier>

Ces commandes fondamentales sont essentielles pour travailler efficacement avec Git. La prochaine section du tutoriel abordera des concepts plus avancés, tels que la gestion des branches, les fusions, et la résolution de conflits.

4. Branches Git

4.1 Création de branches

Pour créer une nouvelle branche :

bash
git branch <nom_de_la_branche>

Pour passer à la nouvelle branche :

bash
git checkout <nom_de_la_branche>

Ou en utilisant la commande git switch (disponible à partir de Git 2.23) :

bash
git switch <nom_de_la_branche>

Pour créer et basculer sur une nouvelle branche en une seule commande :

bash
git checkout -b <nom_de_la_branche>

4.2 Changement entre les branches

Pour passer d'une branche à une autre :

bash
git checkout <nom_de_la_branche>

Ou avec git switch :

bash
git switch <nom_de_la_branche>

4.3 Fusion de branches

Pour fusionner une branche avec la branche actuelle :

  1. Assurez-vous d'être sur la branche de destination :

    bash
    git checkout <branche_de_destination>

  2. Fusionnez la branche spécifiée :

    bash
    git merge <branche_source>

4.4 Résolution des conflits de fusion

En cas de conflit lors de la fusion, Git marquera les fichiers en conflit. Pour résoudre les conflits manuellement :

  1. Ouvrez les fichiers en conflit et résolvez les différences.

  2. Ajoutez les fichiers résolus à l'index :

    bash
    git add <nom_du_fichier>

  3. Terminez la fusion :

    bash
    git merge --continue

4.5 Suppression de branches

Pour supprimer une branche après la fusion :

bash
git branch -d <nom_de_la_branche>

Si la branche n'a pas été fusionnée, utilisez -D pour forcer la suppression :

bash
git branch -D <nom_de_la_branche>

Ce tutoriel a couvert les concepts de base liés aux branches Git. La prochaine section se penchera sur des aspects plus avancés, tels que les remotes, la collaboration, et la gestion des conflits.

5. Remotes Git

5.1 Ajout de remotes

Pour ajouter un dépôt distant (remote) à votre projet :

bash
git remote add <nom_remote> <url_du_dépôt>

5.2 git remote

Affiche la liste des remotes associés au projet :

bash
git remote -v

5.3 git fetch

Récupère les branches distantes et les références sans fusionner les modifications dans le répertoire de travail local :

bash
git fetch <nom_remote>

5.4 git pull

Récupère les modifications du dépôt distant et fusionne automatiquement avec la branche locale en cours :

bash
git pull <nom_remote> <nom_branche>

5.5 git push

Envoie les modifications locales vers le dépôt distant :

bash
git push <nom_remote> <nom_branche>

Si la branche n'existe pas sur le dépôt distant, vous pouvez créer la branche distante en utilisant l'option -u :

bash
git push -u <nom_remote> <nom_branche_locale>

Ces commandes sont essentielles pour travailler avec des dépôts distants et collaborer efficacement avec d'autres contributeurs. La prochaine section abordera la résolution avancée des conflits et d'autres aspects avancés de Git.

6. Collaboration avec Git

6.1 Gestion des branches distantes

Pour lister les branches distantes :

bash
git branch -r

Pour récupérer les dernières modifications des branches distantes :

bash
git fetch --all

6.2 Pull Requests

  1. Création d'une branche pour la fonctionnalité :

    bash
    git checkout -b <nom_de_la_branche>

  2. Ajout, commit, et push des modifications :

    bash
    git add .
git commit -m "Description des modifications"
git push origin <nom_de_la_branche>

  3. Création d'une Pull Request (PR) sur la plateforme de gestion de code (GitHub, GitLab, Bitbucket) :

    • Créez une PR en sélectionnant votre branche et la branche cible.
    • Ajoutez une description détaillée.

  4. Examen par les pairs et modifications :

    • Les collaborateurs peuvent examiner le code, ajouter des commentaires, et demander des modifications.
    • Effectuez les modifications localement et poussez à nouveau.

  5. Fusion de la PR après approbation :

    • Après l'approbation, fusionnez la PR sur la plateforme en ligne.

6.3 Résolution des commentaires et modifications

Si des commentaires ou des modifications sont demandés sur une PR :

  1. Effectuez les modifications localement :

    bash
    git checkout <nom_de_la_branche>

    Faites les modifications, ajoutez et commit.

  2. Push des modifications :

    bash
    git push origin <nom_de_la_branche>

  3. La PR se met à jour automatiquement :

    La PR se mettra à jour avec les nouveaux commits.

6.4 Rebasing vs. Merging

  • Rebasing :

    bash
    git checkout <votre_branche>
git pull --rebase origin <votre_branche>

    Utilisé pour maintenir un historique linéaire. Utile pour nettoyer l'historique avant de fusionner.

  • Merging :

    bash
    git checkout <branche_cible>
git merge <votre_branche>

    Crée un nouveau commit de fusion. Utilisé pour conserver l'historique des branches.

La collaboration avec Git implique une communication efficace et une compréhension des workflows. Assurez-vous de suivre les meilleures pratiques et de bien documenter les modifications. La prochaine section abordera des sujets avancés, comme la gestion des sous-modules et la personnalisation de Git.

7. Gestion des Versions avec Git

7.1 Tags

Les tags sont utilisés pour marquer des points spécifiques dans l'historique des commits, tels que les versions publiques.

Création d'un tag annoté :

bash
git tag -a v1.0 -m "Version 1.0"

Liste des tags :

bash
git tag

Partager un tag avec le dépôt distant :

bash
git push origin v1.0

7.2 git revert

Revert annule un commit spécifique en créant un nouveau commit qui annule les changements du commit précédent.

bash
git revert <commit_id>

7.3 git reset

Reset permet de défaire des commits, mais avec des options différentes.

Reset en douceur (soft reset) :

Annule les commits, mais conserve les modifications dans le répertoire de travail.

bash
git reset --soft <commit_id>

Reset mixte (mixed reset) - Défaut :

Annule les commits et annule les modifications dans l'index (staging).

bash
git reset --mixed <commit_id>

Reset dur (hard reset) :

Annule complètement les commits et les modifications dans le répertoire de travail. Attention, cela peut entraîner une perte de données non commitées.

bash
git reset --hard <commit_id>

7.4 git stash

Stash permet de sauvegarder temporairement les modifications non commitées lorsque vous devez changer de branche.

Stash les modifications :

bash
git stash

Liste des stashes :

bash
git stash list

Appliquer un stash :

bash
git stash apply <nom_stash>

Supprimer un stash :

bash
git stash drop <nom_stash>

Ces commandes de gestion des versions offrent une flexibilité supplémentaire dans la gestion de l'historique des commits et des modifications temporaires. La dernière section de ce tutoriel explorera quelques sujets avancés et des astuces Git.

8. Configuration Avancée de Git

8.1 Alias Git

Les alias sont des raccourcis pour les commandes Git. Ils peuvent être configurés au niveau global ou local.

Création d'un alias global :

bash
git config --global alias.co checkout

Utilisation :

bash
git co <nom_de_branche>

8.2 Hooks Git

Les hooks Git sont des scripts qui peuvent être exécutés automatiquement à certaines étapes du cycle de vie Git, tels que pre-commit, post-commit, pre-push, etc.

Exemple de hook pre-commit :

Créez le fichier .git/hooks/pre-commit et ajoutez le script :

bash
#!/bin/bash

# Exécuter des tests avant chaque commit
npm test

Assurez-vous que le hook est exécutable :

bash
chmod +x .git/hooks/pre-commit

8.3 Configuration globale vs locale

  • Configuration globale :

    bash
    git config --global user.name "Votre Nom"
git config --global user.email "votre@email.com"

    La configuration globale est appliquée à tous les projets de l'utilisateur.

  • Configuration locale (spécifique au projet) :

    bash
    git config user.name "Votre Nom"
git config user.email "votre@email.com"

    La configuration locale est spécifique à un projet et remplace la configuration globale.

Astuces supplémentaires :

  • Autocomplétion Git :

    Activez l'autocomplétion Git en ajoutant ceci à votre fichier de configuration de shell (par exemple, .bashrc ou .zshrc).

    bash
    source /chemin_vers_completion/git-completion.bash

  • Interface graphique Git :

    Utilisez des interfaces graphiques telles que GitKraken, Sourcetree ou GitHub Desktop pour une expérience visuelle de Git.

  • Personnalisation du prompt avec des informations Git :

    Modifiez le prompt de votre terminal pour afficher la branche actuelle, l'état des fichiers, etc.

    bash
    PS1='\[\033[1;32m\]\u@\h\[\033[0m\]:\[\033[1;34m\]\w\[\033[0m\]$(parse_git_branch)\$ '

Ce tutoriel couvre des aspects avancés de la configuration Git, tels que les alias, les hooks et la gestion des configurations globales et locales. Avec ces connaissances, vous serez en mesure d'optimiser davantage votre utilisation de Git.

9. Travail en Équipe avec Git

9.1 Conventions de Nommage des Branches

Adopter des conventions de nommage des branches facilite la compréhension du flux de travail dans un projet. Voici quelques recommandations :

  • Master/Main : Branche principale du projet.
  • Feature : Pour le développement de nouvelles fonctionnalités.
    • Exemple : feature/nom_de_la_fonctionnalite
  • Bugfix : Pour la correction de bugs.
    • Exemple : bugfix/nom_du_bug
  • Hotfix : Pour les corrections d'urgence sur la production.
    • Exemple : hotfix/nom_de_la_correction
  • Release : Préparation des versions à déployer.
    • Exemple : release/version_x.x.x

9.2 Bonnes Pratiques de Gestion de Projet

  • Documentation : Maintenez une documentation claire sur l'architecture, les conventions et le flux de travail.

  • Revues de Code : Encouragez les revues de code pour garantir la qualité du code et le partage des connaissances.

  • Issues et Tickets : Utilisez un système de suivi des problèmes pour signaler les bugs, les fonctionnalités à ajouter, et les tâches à effectuer.

  • Intégration Continue : Configurez des outils d'intégration continue pour automatiser les tests et la construction du projet.

  • Environnements de Développement Similaires : Assurez-vous que les membres de l'équipe utilisent des environnements de développement similaires.

9.3 Stratégies de Workflow Git

9.3.1 Git Flow

Le Git Flow est un modèle de flux de travail populaire avec les branches suivantes :

  • Master/Main : Pour les versions stables du projet.
  • Develop : Branche de développement.
  • Feature : Pour le développement de nouvelles fonctionnalités.
  • Release : Préparation des versions.
  • Hotfix : Corrections d'urgence.

9.3.2 GitHub Flow

Le GitHub Flow est un modèle plus simple avec une seule branche principale :

  • Main : Branche principale du projet.
  • Feature Branches : Création de branches pour chaque nouvelle fonctionnalité.
  • Pull Requests : Intégration des fonctionnalités par le biais de pull requests.

9.3.3 GitLab Flow

Le GitLab Flow est similaire au GitHub Flow avec quelques différences, notamment une branche "staging" pour les tests avant la production.

  • Master : Branche principale du projet.
  • Feature Branches : Développement de nouvelles fonctionnalités.
  • Staging : Test avant la production.
  • Production : Branche déployée en production.

Choisissez la stratégie qui convient le mieux à votre projet et à votre équipe. L'essentiel est d'adopter une approche cohérente pour garantir la stabilité et la collaboration efficace.

10. Dépannage avec Git

10.1 Erreurs Fréquentes et Solutions

10.1.1 Problème : Conflits de Fusion

  • Solution : Utilisez git status pour identifier les fichiers en conflit. Ouvrez chaque fichier en conflit, résolvez les conflits manuellement, puis exécutez git add <fichier> suivi de git merge --continue.

10.1.2 Problème : Référentiel Corrompu

  • Solution : Exécutez git fsck --full pour vérifier l'intégrité du référentiel. Si des objets sont manquants, essayez de les récupérer à partir d'une copie de sauvegarde ou avec des commandes spécifiques.

10.1.3 Problème : Oubli de Committre Avant le Changement de Branche

  • Solution : Utilisez git stash pour sauvegarder temporairement vos modifications, puis changez de branche. Après le changement de branche, utilisez git stash apply pour réappliquer vos modifications.

10.2 Commandes de Dépannage Utiles

10.2.1 git reflog

Affiche l'historique des références Git, y compris les changements de branches et les actions spécifiques à Git.

bash
git reflog

10.2.2 git cherry-pick

Applique un commit spécifique à une branche différente.

bash
git cherry-pick <commit_id>

10.2.3 git bisect

Aide à trouver le commit qui a introduit un bogue en effectuant une recherche binaire.

bash
git bisect start
git bisect bad
git bisect good <commit_id>

10.2.4 git clean

Supprime les fichiers non suivis dans le répertoire de travail.

bash
git clean -n  # Affiche les fichiers à supprimer
git clean -f  # Supprime réellement les fichiers

10.2.5 git revert

Annule les modifications d'un commit en créant un nouveau commit.

bash
git revert <commit_id>

Ces commandes sont utiles pour résoudre certains des problèmes courants que vous pourriez rencontrer lors de l'utilisation de Git. Si vous rencontrez des difficultés, n'oubliez pas de consulter la documentation officielle de Git et la communauté pour obtenir de l'aide supplémentaire.
