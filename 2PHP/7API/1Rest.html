6.1 Concepts de base de REST

6.1.1 REST (Representational State Transfer)

REST est un style d'architecture logicielle qui repose sur des principes fondamentaux. Ces principes permettent de concevoir des systèmes web évolutifs et interopérables.

6.1.2 Ressources

Les ressources sont des entités que vous pouvez manipuler via l'API REST. Elles peuvent être des objets, des données, ou même des services.

6.1.3 URI (Uniform Resource Identifier)

Chaque ressource est identifiée par une URI unique. Les URI doivent être descriptives et permettre de comprendre la nature de la ressource.

6.1.4 Représentation

Les ressources peuvent avoir différentes représentations (JSON, XML, HTML, etc.). Les clients et serveurs peuvent négocier le format via les en-têtes HTTP.

6.1.5 Opérations HTTP

REST utilise les méthodes HTTP (GET, POST, PUT, DELETE, etc.) pour effectuer des opérations sur les ressources. Par exemple, GET pour récupérer une ressource, POST pour créer une nouvelle ressource, PUT pour mettre à jour une ressource, DELETE pour supprimer une ressource, etc.

6.2 Principes de conception RESTful

6.2.1 Stateless (Sans état)

Chaque requête du client au serveur doit contenir toutes les informations nécessaires pour comprendre et traiter la demande. Aucun état de session n'est stocké sur le serveur entre les requêtes.

6.2.2 Interface uniforme

L'interface entre les composants doit être uniforme, ce qui signifie que les conventions doivent être respectées pour simplifier l'interaction.

6.2.3 Découplage client-serveur

Le client et le serveur sont des entités distinctes et doivent évoluer indépendamment. Le serveur peut évoluer sans affecter le client et vice versa.

6.2.4 Système en couches

Le système peut être composé de plusieurs couches (par exemple, un serveur, une base de données, etc.), chaque couche ne connaissant que celle juste en dessous ou au-dessus d'elle.

6.3 Avantages de REST

6.3.1 Scalabilité

Grâce à l'architecture sans état et au découplage client-serveur, les systèmes REST sont facilement scalables.

6.3.2 Interopérabilité

Les services REST peuvent être consommés par des clients écrits dans différents langages, car ils utilisent des standards ouverts tels que HTTP et JSON.

6.3.3 Simplicité

L'architecture REST est simple et utilise les normes du web existantes, ce qui facilite la compréhension et l'implémentation.

6.3.4 Évolutivité

Les services REST peuvent évoluer de manière indépendante, permettant des mises à jour sans impact sur les autres parties du système.

En suivant ces principes, vous pouvez concevoir des API RESTful robustes et flexibles en PHP. Assurez-vous de respecter les normes et de documenter clairement votre API pour simplifier son utilisation par d'autres développeurs.

8.1 Authentification basique

8.1.1 Utilisation de l'en-tête Authorization

L'authentification basique se fait souvent en utilisant l'en-tête Authorization avec le type Basic.

php
// Extrait les informations d'identification de l'en-tête
$authHeader = $_SERVER['HTTP_AUTHORIZATION'];
list($type, $credentials) = explode(" ", $authHeader);

// Décodage des informations d'identification
list($username, $password) = explode(":", base64_decode($credentials));

// Vérification des informations d'identification
if ($username === 'utilisateur' && $password === 'motdepasse') {
    // Authentification réussie
} else {
    // Authentification échouée
    header('HTTP/1.1 401 Unauthorized');
    echo 'Authentification échouée';
    exit();
}

8.2 Utilisation de tokens d'accès

8.2.1 Génération d'un token d'accès

Utilisez un mécanisme tel que JSON Web Tokens (JWT) pour générer des tokens d'accès.

bash
composer require firebase/php-jwt

php
use Firebase\JWT\JWT;

$secretKey = 'votre_clé_secrète';
$payload = array(
    "user_id" => 1,
    "username" => "utilisateur",
    "role" => "admin"
);

$token = JWT::encode($payload, $secretKey, 'HS256');

8.2.2 Validation du token

Dans vos points de terminaison, vérifiez la validité du token.

php
use Firebase\JWT\JWT;

$secretKey = 'votre_clé_secrète';
$token = $_SERVER['HTTP_AUTHORIZATION'];

try {
    $decoded = JWT::decode($token, $secretKey, array('HS256'));
    // Token valide, continuez avec la logique de l'API
} catch (Exception $e) {
    header('HTTP/1.1 401 Unauthorized');
    echo 'Token invalide';
    exit();
}
