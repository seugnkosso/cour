1.1 Principes fondamentaux de la POO en PHP

1.1.1 Classes et Objets

La POO repose sur le concept de classes et d'objets. Une classe est un modèle pour créer des objets, qui sont des instances de cette classe.

php
class Personne {
    public $nom;
    public $age;

    public function parler() {
        echo "Bonjour, je m'appelle $this->nom et j'ai $this->age ans.";
    }
}

// Instanciation de l'objet
$personne1 = new Personne();
$personne1->nom = "John Doe";
$personne1->age = 30;

// Appel de la méthode
$personne1->parler();

1.1.2 Encapsulation

Encapsulation signifie cacher la complexité interne d'une classe et ne montrer que ce qui est nécessaire à l'extérieur.

php
class CompteBancaire {
    private $solde;

    public function getSolde() {
        return $this->solde;
    }

    public function deposer($montant) {
        $this->solde += $montant;
    }

    public function retirer($montant) {
        if ($montant <= $this->solde) {
            $this->solde -= $montant;
        } else {
            echo "Solde insuffisant.";
        }
    }
}

1.1.3 Héritage

L'héritage permet à une classe d'utiliser les propriétés et méthodes d'une autre classe. Cela favorise la réutilisation du code.

php
class Etudiant extends Personne {
    public $niveau;

    public function etudier() {
        echo "$this->nom étudie au niveau $this->niveau.";
    }
}

1.1.4 Polymorphisme

Le polymorphisme permet à des objets de différentes classes de répondre de manière uniforme à une même méthode.

php
interface Forme {
    public function calculerSurface();
}

class Carre implements Forme {
    public $cote;

    public function calculerSurface() {
        return $this->cote * $this->cote;
    }
}

class Cercle implements Forme {
    public $rayon;

    public function calculerSurface() {
        return pi() * $this->rayon * $this->rayon;
    }
}

1.2 Avantages de la POO en PHP

1.2.1 Modularité

La POO favorise la création de modules réutilisables, ce qui facilite la maintenance et l'évolutivité du code.

1.2.2 Encapsulation et Sécurité

En encapsulant les propriétés et les méthodes, la POO permet de protéger le code contre les accès non autorisés et d'éviter les interférences entre les différentes parties du programme.

1.2.3 Héritage et Réutilisabilité

L'héritage permet de créer de nouvelles classes basées sur des classes existantes, favorisant ainsi la réutilisabilité du code.

1.2.4 Polymorphisme et Flexibilité

Le polymorphisme permet d'utiliser une interface commune pour des objets de classes différentes, offrant ainsi une flexibilité dans la conception du code.

En adoptant la POO en PHP, vous pouvez améliorer la structure, la maintenance et la flexibilité de votre code, facilitant ainsi le développement d'applications robustes et extensibles.

3.1 Définition d'une interface

Une interface est un contrat définissant un ensemble de méthodes qu'une classe doit implémenter. Les interfaces ne contiennent que des signatures de méthodes, sans implémentation.

php
interface Forme {
    public function calculerSurface();
}

3.2 Implémentation d'une interface

Une classe peut implémenter une ou plusieurs interfaces en utilisant le mot-clé implements.

php
class Carre implements Forme {
    public $cote;

    public function calculerSurface() {
        return $this->cote * $this->cote;
    }
}

3.3 Constantes dans les interfaces

Les interfaces peuvent également contenir des constantes.

php
interface Couleurs {
    const ROUGE = "FF0000";
    const VERT = "00FF00";
}

3.4 Méthodes abstraites dans les interfaces

Toutes les méthodes dans une interface sont par défaut abstraites et publiques. Une classe implémentant une interface doit fournir une implémentation pour chaque méthode de l'interface.

php
interface Animal {
    public function manger();
    public function dormir();
}

class Chat implements Animal {
    public function manger() {
        echo "Le chat mange.";
    }

    public function dormir() {
        echo "Le chat dort.";
    }
}

3.5 Interface vs. Classe abstraite

Les interfaces et les classes abstraites sont similaires, mais il y a des différences. Une classe abstraite peut contenir des méthodes concrètes et des propriétés, tandis qu'une interface ne contient que des méthodes abstraites et des constantes. Une classe peut étendre une seule classe abstraite, mais elle peut implémenter plusieurs interfaces.

3.6 Héritage multiple avec les interfaces

Une classe peut implémenter plusieurs interfaces, permettant ainsi l'héritage multiple.

php
interface Volant {
    public function voler();
}

interface Nageant {
    public function nager();
}

class AnimalVolant implements Volant {
    public function voler() {
        echo "L'animal vole.";
    }
}

class Oiseau implements Volant, Nageant {
    public function voler() {
        echo "L'oiseau vole.";
    }

    public function nager() {
        echo "L'oiseau nage.";
    }
}

En utilisant les interfaces en PHP, vous pouvez créer des contrats clairs entre vos classes, favorisant la réutilisabilité du code et permettant une meilleure organisation de votre architecture logicielle. N'hésitez pas à pratiquer ces concepts pour les maîtriser efficacement.

4.1 Définition d'un trait

Un trait est une collection de méthodes regroupées dans un bloc de code réutilisable. Les traits sont similaires aux classes, mais ils ne peuvent pas être instanciés eux-mêmes.

php
trait Enregistrement {
    public function enregistrer() {

Un trait peut être utilisé dans une classe en utilisant le mot-clé use. Les méthodes du trait deviennent alors disponibles dans la classe.

4.3 Résolution des conflits

Si deux traits utilisés par une classe ont une méthode du même nom, un conflit se produit. Ce conflit peut être résolu en utilisant le mot-clé insteadof ou as.

Résolution avec insteadof

php
trait TraitA {
    public function direBonjour() {
        echo "Bonjour de TraitA";
    }
}

trait TraitB {
    public function direBonjour() {
        echo "Bonjour de TraitB";
    }
}

class MaClasse {
    use TraitA, TraitB {
        TraitA::direBonjour insteadof TraitB;
    }
}

Résolution avec as

php
trait TraitA {
    public function direBonjour() {
        echo "Bonjour de TraitA";
    }
}

trait TraitB {
    public function direBonjour() {
        echo "Bonjour de TraitB";
    }
}

class MaClasse {
    use TraitA, TraitB {
        TraitA::direBonjour as direBonjourTraitA;
        TraitB::direBonjour as direBonjourTraitB;
    }
}

En utilisant des traits, vous pouvez réutiliser du code de manière horizontale sans utiliser l'héritage, ce qui est particulièrement utile lorsque vous ne pouvez pas utiliser l'héritage multiple ou lorsque les classes n'ont pas de relation d'héritage logique. N'oubliez pas de bien gérer les conflits si vous utilisez plusieurs traits dans une même classe.

5.1 Surcharge (Overloading)

La surcharge (overloading) en PHP permet à une classe de définir plusieurs méthodes ayant le même nom mais avec des signatures différentes. Il existe deux types de surcharge : la surcharge de méthodes et la surcharge de propriétés.

5.1.1 Surcharge de méthodes

La surcharge de méthodes se produit lorsque des méthodes du même nom sont définies dans une classe avec des signatures différentes.

php
class Calculatrice {
    public function additionner($a, $b) {
        return $a + $b;
    }

    public function additionnerTrois($a, $b, $c) {
        return $a + $b + $c;
    }
}

5.1.2 Surcharge de propriétés

La surcharge de propriétés se produit lorsque des propriétés du même nom sont définies dans une classe avec des visibilités différentes.

php
class Exemple {
    public $propriete;

    public function __get($nom) {
        echo "Propriété '$nom' inaccessible.";
    }

    public function __set($nom, $valeur) {
        echo "Propriété '$nom' inexistante ou inaccessible.";
    }
}

5.2 Redéfinition (Overriding)

La redéfinition (overriding) se produit lorsqu'une classe dérivée fournit une implémentation spécifique pour une méthode déjà définie dans la classe de base. Cela permet à une classe enfant de fournir une version différente d'une méthode déjà définie dans sa classe parent.

php
class Animal {
    public function parler() {
        echo "Animal parle.";
    }
}

class Chat extends Animal {
    public function parler() {
        echo "Le chat miaule.";
    }
}

En utilisant la surcharge et la redéfinition, vous pouvez créer des classes qui partagent une interface commune tout en permettant à chaque classe d'avoir une implémentation spécifique. Cela favorise la flexibilité et la réutilisabilité du code en rendant vos classes interchangeables dans différentes situations.

7.1 Singleton

Le Singleton garantit qu'une classe n'a qu'une seule instance et fournit un point d'accès global à cette instance.

php
class Singleton {
    private static $instance;

    private function __construct() {}

    public static function getInstance() {
        if (!isset(self::$instance)) {
            self::$instance = new self();
        }
        return self::$instance;
    }
}

7.2 Fabrique (Factory)

La Fabrique fournit une interface pour créer des objets dans une classe parente, mais permet à ses sous-classes de modifier le type d'objets qui seront créés.

php
interface Fabrique {
    public function creerProduit();
}

class FabriqueA implements Fabrique {
    public function creerProduit() {
        return new ProduitA();
    }
}

class FabriqueB implements Fabrique {
    public function creerProduit() {
        return new ProduitB();
    }
}

7.3 Observateur (Observer)

L'Observateur permet à un objet, appelé sujet, de maintenir une liste de ses dépendances, appelées observateurs, qui sont informées des changements d'état du sujet.

php
interface Observateur {
    public function mettreAJour($etat);
}

class Sujet {
    private $observateurs = [];

    public function ajouterObservateur(Observateur $observateur) {
        $this->observateurs[] = $observateur;
    }

    public function notifierObservateurs($etat) {
        foreach ($this->observateurs as $observateur) {
            $observateur->mettreAJour($etat);
        }
    }
}

7.5 Modèle-Vue-Contrôleur (MVC)

Le Modèle-Vue-Contrôleur (MVC) sépare les données (Modèle), la présentation (Vue) et le traitement des actions de l'utilisateur (Contrôleur) dans une application.

php
class Modele {
    private $donnees;

    public function setDonnees($donnees) {
        $this->donnees = $donnees;
    }

    public function getDonnees() {
        return $this->donnees;
    }
}

class Vue {
    public function afficherDonnees($donnees) {
        echo "Affichage des données : $donnees";
    }
}

class Controleur {
    private $modele;
    private $vue;

    public function __construct(Modele $modele, Vue $vue) {
        $this->modele = $modele;
        $this->vue = $vue;
    }

    public function traiterRequete() {
        // Logique métier
        $donnees = "Données de l'application";
        
        // Mise à jour du modèle
        $this->modele->setDonnees($donnees);

        // Affichage par la vue
        $this->vue->afficherDonnees($this->modele->getDonnees());
    }
}

// Utilisation
$modele = new Modele();
$vue = new Vue();
$controleur = new Controleur($modele, $vue);
$controleur->traiterRequete();

8.2 PSR-4 Autoloading Standard

PSR-4 est une spécification définie par la PHP-FIG (PHP Framework Interop Group) pour l'autoloading des classes. Elle recommande une structure de répertoires et un format de noms de classes pour simplifier l'autoloading.

8.2.1 Structure de répertoires

css

  MonProjet/
    MaClasse.php
tests/
  MonProjetTest/
    MaClasseTest.php

php
// Définition de la fonction d'autoloading conforme à PSR-4
function monAutoloader($nomClasse) {
    // Convertir les séparateurs de noms de classe en séparateurs de fichiers
    $cheminClasse = str_replace('\\', DIRECTORY_SEPARATOR, $nomClasse);

    // Construire le chemin complet vers le fichier de classe
    $cheminFichier = __DIR__ . '/src/' . $cheminClasse . '.php';

    if (file_exists($cheminFichier)) {
        include $cheminFichier;
    }

// Utilisation de la classe conforme à PSR-4

Les méthodes magiques sont des méthodes spéciales dans PHP qui sont préfixées par deux caractères de soulignement (__). Elles permettent de définir un comportement particulier pour certaines actions sur les objets.

9.3.1 __toString

La méthode magique __toString permet de définir comment l'objet doit être converti en chaîne de caractères lorsqu'il est utilisé dans un contexte de chaîne.

php
class Personne {
    public $nom;

    public function __construct($nom) {
        $this->nom = $nom;
    }

    public function __toString() {
        return "Personne: " . $this->nom;
    }
}

$personne = new Personne("John Doe");
echo $personne;  // Affiche "Personne: John Doe"

9.3.2 __get et __set

Les méthodes magiques __get et __set permettent de définir le comportement lors de l'accès à des propriétés qui ne sont pas accessibles directement.

php
class Voiture {
    private $proprietes = [];

    public function __get($nom) {
        if (isset($this->proprietes[$nom])) {
            return $this->proprietes[$nom];
        } else {
            echo "La propriété '$nom' n'existe pas.";
        }
    }

    public function __set($nom, $valeur) {
        $this->proprietes[$nom] = $valeur;
    }
}

$voiture = new Voiture();
$voiture->couleur = "rouge";
echo $voiture->couleur;  // Affiche "rouge"

13.1 Principes SOLID

Les principes SOLID sont un ensemble de cinq principes de conception de logiciels qui visent à rendre le code plus compréhensible, flexible et maintenable.

13.1.1 SRP - Principe de responsabilité unique

Une classe devrait avoir une seule raison de changer, c'est-à-dire qu'elle devrait avoir une seule responsabilité.

13.1.2 OCP - Principe d'ouverture/fermeture

Une classe doit être ouverte à l'extension mais fermée à la modification. Cela signifie que vous devriez pouvoir ajouter de nouvelles fonctionnalités à une classe sans modifier son code existant.

13.1.3 LSP - Principe de substitution de Liskov

Les objets d'une classe dérivée devraient pouvoir être utilisés à la place des objets de la classe de base sans affecter la cohérence du programme.

13.1.4 ISP - Principe d'inversion des dépendances

Les modules de haut niveau ne doivent pas dépendre de modules de bas niveau. Les deux devraient dépendre d'abstractions. Les abstractions ne doivent pas dépendre des détails, mais les détails doivent dépendre des abstractions.

13.1.5 DIP - Principe de ségrégation des interfaces

Une classe ne doit pas être forcée de mettre en œuvre des interfaces dont elle n'a pas besoin.

13.2 Cohésion et couplage

13.2.1 Cohésion

La cohésion mesure le degré d'interdépendance entre les membres d'une classe. Une classe cohérente a des membres qui travaillent ensemble pour atteindre un objectif commun.

13.2.2 Couplage

Le couplage mesure le niveau d'interdépendance entre deux classes. Un couplage faible est souhaitable, car il rend le code plus modulaire et facile à maintenir.

13.3 Conventions de nommage PSR

Les PSR (PHP Standard Recommendations) sont un ensemble de recommandations de codage pour la communauté PHP. Les PSR-1, PSR-2 et PSR-4 sont particulièrement pertinents pour les bonnes pratiques de codage orienté objet.

13.3.1 PSR-1 : Basic Coding Standard

  • Les noms de classes doivent être écrits en PascalCase.
  • Les noms de méthodes et de propriétés doivent être écrits en camelCase.
  • Les constantes doivent être entièrement en majuscules avec des mots séparés par des underscores.

13.3.2 PSR-2 : Coding Style Guide

PSR-2 étend PSR-1 et propose des règles plus détaillées pour le style de code.

13.3.3 PSR-4 : Autoloading Standard

PSR-4 définit un standard pour l'autoloading des classes, simplifiant ainsi la gestion des dépendances dans les projets PHP.

En suivant ces principes et conventions, vous contribuez à la création de code orienté objet plus lisible, extensible et maintenable. Adopter les PSR favorise également une cohérence au sein de la communauté PHP. N'oubliez pas que les bonnes pratiques de codage évoluent avec le temps, alors assurez-vous de rester à jour avec les dernières recommandations et tendances.
