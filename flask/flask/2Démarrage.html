Une Application Minime¶

Une application Flask minimale ressemble à ceci:

from flask import Flask

app = Flask(__name__)

@app.route("/")
def hello_world():
    return "<p>Hello, World!</p>"

Alors, qu'est-ce que ce code a fait?

  1. Nous avons d'abord importé le Flask classe. Une instance de cette classe sera notre application WSGI.

  2. Ensuite, nous créons une instance de cette classe. Le premier argument est le nom de l'application’s module or package. __name__ est un raccourci pratique pour cela qui est approprié pour la plupart des cas. Ceci est nécessaire pour que Flask sache où chercher des ressources telles que comme modèles et fichiers statiques.

  3. Nous utilisons ensuite le route() décorateur à dire à Flask quelle URL devrait déclencher notre fonction.

  4. La fonction renvoie le message que nous voulons afficher dans user’s navigateur. Le type de contenu par défaut est HTML, donc HTML dans la chaîne sera rendu par le navigateur.

Enregistrez-le comme hello.py ou quelque chose de similaire. Assurez-vous de ne pas appeler votre application flask.py parce que cela serait en conflit avec Flask lui-même.

Pour exécuter l'application, utilisez le flask commande ou . Vous devez indiquer au flacon où se trouve votre demande est avec le option.python -m flask--app

$ flask --app hello run
 * Serving Flask app 'hello'
 * Running on http://127.0.0.1:5000 (Press CTRL+C to quit)

Comportement Découverte Application

En tant que raccourci, si le fichier est nommé app.py ou wsgi.py, vous don’t doivent utiliser --app. Voir Interface Ligne Commande pour plus de détails.

Cela lance un serveur intégré très simple, qui est assez bon pour test mais probablement pas ce que vous voulez utiliser en production. Pour options de déploiement voir Déploiement en Production.

Maintenant, dirigez-vous vers http://127.0.0.1:5000/, et tu devrais voir ton bonjour salutation du monde.

Serveur Visible Extérieurement

Si vous exécutez le serveur, vous remarquerez que le serveur est uniquement accessible depuis votre propre ordinateur, pas depuis un autre réseau. C'est le par défaut, car en mode de débogage, un utilisateur de l'application peut exécuter code Python arbitraire sur votre ordinateur.

Si le débogueur est désactivé ou faites confiance aux utilisateurs de votre réseau, vous pouvez rendre le serveur accessible au public simplement en ajoutant --host=0.0.0.0 en ligne de commande:

$ flask run --host=0.0.0.0

Cela indique à votre système d'exploitation d'écouter sur toutes les adresses IP publiques.

HTML Échapper¶

Lors du retour de HTML (le type de réponse par défaut dans Flask) les valeurs fournies par l'utilisateur dans la sortie doivent être échappées pour protéger des attaques d'injection. Modèles HTML rendus avec Jinja, introduit plus tard, cela se fera automatiquement.

escape(), montré ici, peut être utilisé manuellement. Il est omis dans la plupart des exemples pour la brièveté, mais vous devriez toujours être conscient de comment vous’re en utilisant des données non fiables.

from markupsafe import escape

@app.route("/<name>")
def hello(name):
    return f"Hello, {escape(name)}!"

Routage¶

Les applications Web modernes utilisent des URL significatives pour aider les utilisateurs. Les utilisateurs sont plus susceptibles d'aimer une page et de revenir si la page utilise une URL significative qu'ils peuvent rappelez-vous et utilisez pour visiter directement une page.

Utilisez le route() décorateur pour lier une fonction à une URL.

@app.route('/')
def index():
    return 'Index Page'

@app.route('/hello')
def hello():
    return 'Hello, World'

Vous pouvez faire plus! Vous pouvez rendre des parties de l'URL dynamiques et en joindre plusieurs règles pour une fonction.

Variable Règles¶

Vous pouvez ajouter des sections variables à une URL en marquant les sections avec <variable_name>. Votre fonction reçoit alors le <variable_name> comme argument de mot-clé. En option, vous pouvez utiliser un convertisseur pour spécifier le type de l'argument comme <converter:variable_name>.

from markupsafe import escape

@app.route('/user/<username>')
def show_user_profile(username):
    # show the user profile for that user
    return f'User {escape(username)}'

@app.route('/post/<int:post_id>')
def show_post(post_id):
    # show the post with the given id, the id is an integer
    return f'Post {post_id}'

@app.route('/path/<path:subpath>')
def show_subpath(subpath):
    # show the subpath after /path/
    return f'Subpath {escape(subpath)}'

Convertisseur types:

string

(par défaut) accepte tout texte sans barre oblique

int

accepte les entiers positifs

float

accepte les valeurs positives en virgule flottante

path

comme string mais accepte aussi les barres obliques

uuid

accepte les chaînes UUID

URLs uniques / Comportement de redirection¶

Les deux règles suivantes diffèrent dans leur utilisation d'une barre oblique.

@app.route('/projects/')
def projects():
    return 'The project page'

@app.route('/about')
def about():
    return 'The about page'

L'URL canonique pour le projects endpoint a une barre oblique. It’s similaire à un dossier dans un système de fichiers. Si vous accédez à l'URL sans une barre oblique (/projects), Flask vous redirige vers l'URL canonique avec la barre oblique (/projects/').

L'URL canonique pour le about endpoint n'a pas de fin barre. It’s similar to the pathname of a file. Accéder à l'URL avec un barre oblique (/about/) produit une erreur 404 “Not Found”. Cela aide gardez les URL uniques pour ces ressources, ce qui aide les moteurs de recherche à éviter indexer deux fois la même page.

URL Building¶

Pour créer une URL vers une fonction spécifique, utilisez le url_for() fonction. Il accepte le nom de la fonction comme premier argument et n'importe quel nombre de arguments de mots-clés, chacun correspondant à une partie variable de la règle URL. Les parties de variables inconnues sont ajoutées à l'URL en tant que paramètres de requête.

Pourquoi voudriez-vous construire des URL en utilisant la fonction d'inversion d'URL url_for() au lieu de les coder en dur dans vos modèles?

  1. L'inversion est souvent plus descriptive que le codage dur des URL.

  2. Vous pouvez modifier vos URL en une seule fois au lieu de vous en souvenir modifier manuellement les URL codées en dur.

  3. La construction d'URL gère l'échappement des caractères spéciaux de manière transparente.

  4. Les chemins générés sont toujours absolus, évitant les comportements inattendus des chemins relatifs dans les navigateurs.

  5. Si votre application est placée en dehors de la racine de l'URL, par exemple /myapplication au lieu de /, url_for() correctement gère cela pour vous.

Par exemple, ici, nous utilisons le test_request_context() méthode essayer url_for(). test_request_context() indique à Flask de se comporter comme s'il était en train de traiter une demande même lorsque nous utilisons un Python shell shell python. Voir Contexte Locaux.

from flask import url_for

@app.route('/')
def index():
    return 'index'

@app.route('/login')
def login():
    return 'login'

@app.route('/user/<username>')
def profile(username):
    return f'{username}\'s profile'

with app.test_request_context():
    print(url_for('index'))
    print(url_for('login'))
    print(url_for('login', next='/'))
    print(url_for('profile', username='John Doe'))

HTTP Méthodes¶

Les applications Web utilisent différentes méthodes HTTP pour accéder aux URL. Vous devriez familiarisez-vous avec les méthodes HTTP lorsque vous travaillez avec Flask. Par défaut, un itinéraire ne répond qu'à GET demandes. Vous pouvez utiliser le methods argument de la route() décorateur pour gérer différentes méthodes HTTP.

from flask import request

@app.route('/login', methods=['GET', 'POST'])
def login():
    if request.method == 'POST':
        return do_the_login()
    else:
        return show_the_login_form()

L'exemple ci-dessus conserve toutes les méthodes pour la route dans une fonction, ce qui peut être utile si chaque partie utilise des données communes.

Vous pouvez également séparer les vues pour différentes méthodes en différentes options fonctions. Flask fournit un raccourci pour décorer ces itinéraires avec get(), post(), etc pour chacun méthode HTTP commune.

@app.get('/login')
def login_get():
    return show_the_login_form()

@app.post('/login')
def login_post():
    return do_the_login()

Si GET est présent, Flask ajoute automatiquement le support pour le HEAD méthode et poignées HEAD demandes selon le HTTP RFC. De même, OPTIONS est automatiquement mis en œuvre pour vous.

Static Fichiers¶

Les applications Web dynamiques ont également besoin de fichiers statiques. That’s usually where les fichiers CSS et JavaScript proviennent de. Idéalement, votre serveur web est configuré pour les servir pour vous, mais pendant le développement Flask peut le faire aussi. Créez simplement un dossier appelé static dans votre colis ou à côté de votre module et il sera disponible à /static sur l'application.

Pour générer des URL pour les fichiers statiques, utilisez le 'static' nom du point final:

url_for('static', filename='style.css')

Le fichier doit être stocké sur le système de fichiers comme static/style.css.

Modèles de rendu¶

Générer du HTML à partir de Python n'est pas amusant, et en fait joli encombrant parce que vous devez faire le HTML s'échappant sur votre propre pour garder l'application sécurisée. En raison de ce Flask configure le Jinja2 moteur de modèle pour vous automatiquement.

Les modèles peuvent être utilisés pour générer tout type de fichier texte. Pour les applications web, you’s générez principalement des pages HTML, mais vous pouvez également générer du markdown, du texte brut e-mails, et tout le reste.

Pour une référence à HTML, CSS, et d'autres API Web, utilisez le MDN Web Docs.

Pour rendre un modèle, vous pouvez utiliser le render_template() méthode. Tout ce que vous avez à faire est de fournir le nom du modèle variables que vous souhaitez transmettre au moteur de modèle sous forme d'arguments de mots clés. Here’s un exemple simple de la façon de rendre un modèle:

from flask import render_template

@app.route('/hello/')
@app.route('/hello/<name>')
def hello(name=None):
    return render_template('hello.html', name=name)

Flask recherchera des modèles dans le templates dossier. Donc si votre application est un module, ce dossier est à côté de ce module, si elle package it’s réellement dans votre package:

Cas 1: un module:

/application.py
/templates
    /hello.html

Cas 2: un paquet:

/application
    /__init__.py
    /templates
        /hello.html

Pour les modèles, vous pouvez utiliser toute la puissance des modèles Jinja2. Tête au-dessus au fonctionnaire Jinja2 Modèle Documentation pour plus d'informations.

Voici un exemple de modèle:

<!doctype html>
<title>Hello from Flask</title>
{% if name %}
  <h1>Hello {{ name }}!</h1>
{% else %}
  <h1>Hello, World!</h1>
{% endif %}

Les modèles sont particulièrement utiles si l'héritage est utilisé. Si vous voulez sachez comment cela fonctionne, voyez Modèle Héritage. Fondamentalement l'héritage de modèle permet de conserver certains éléments sur chacun page (comme en-tête, navigation et pied de page).

L'échappement automatique est activé, donc si name contient du HTML, il sera échappé automatiquement. Si vous pouvez faire confiance à une variable et que vous savez que ce sera safe HTML (par exemple parce qu'il provient d'un module qui convertit wiki balisage en HTML) vous pouvez le marquer comme sûr en utilisant le Markup en classe ou en utilisant le |safe filtre dans le modèle. Contexte Locaux¶

Informations d'Initié

Si vous voulez comprendre comment cela fonctionne et comment vous pouvez mettre en œuvre tests avec des sections locales contextuelles, lisez cette section, sinon sautez-la.

Certains objets dans Flask sont des objets globaux, mais pas du genre habituel. Ces objets sont en fait des proxys d'objets locaux à un spécifique contexte. Quelle bouchée. Mais c'est en fait assez facile à comprendre.

Imaginez le contexte comme fil de manipulation. Une demande entre et le le serveur Web décide d'engendrer un nouveau thread (ou autre chose, le l'objet sous-jacent est capable de traiter avec d'autres systèmes de concurrence que des fils). Lorsque Flask commence sa requête interne en la gérant calcule que le thread courant est le contexte actif et lie le application actuelle et les environnements WSGI à ce contexte (thread). Il le fait de manière intelligente afin qu'une application puisse en invoquer une autre application sans rupture.

Alors qu'est-ce que cela signifie pour vous? Fondamentalement, vous pouvez complètement ignorer cela c'est le cas à moins que vous ne fassiez quelque chose comme des tests unitaires. Vous remarquera que le code qui dépend d'un objet de requête se brisera soudainement parce qu'il n'y a pas d'objet de requête. La solution consiste à créer une requête objectez-vous et reliez-le au contexte. La solution la plus simple pour le test unitaire consiste à utiliser test_request_context() gestionnaire de contexte. En combinaison avec le with déclaration qu'il va lier un tester la demande afin que vous puissiez interagir avec elle. Voici un exemple:

from flask import request

with app.test_request_context('/hello', method='POST'):
    # now you can do something with the request until the
    # end of the with block, such as basic assertions:
    assert request.path == '/hello'
    assert request.method == 'POST'

L'autre possibilité est de passer tout un environnement WSGI à la request_context() méthode:

with app.request_context(environ):
    assert request.method == 'POST'

Téléchargements de fichiers¶

Vous pouvez facilement gérer les fichiers téléchargés avec Flask. Assurez-vous simplement de ne pas oublier de régler le enctype="multipart/form-data" attribut sur votre HTML forme, sinon le navigateur ne transmettra pas du tout vos fichiers.

Les fichiers téléchargés sont stockés en mémoire ou à un emplacement temporaire système de fichiers. Vous pouvez accéder à ces fichiers en regardant le files attribut sur l'objet de requête. Chacun le fichier téléchargé est stocké dans ce dictionnaire. Il se comporte comme un standard Python file objet, mais il a aussi un save() méthode que permet de stocker ce fichier sur le système de fichiers du serveur. Voici un exemple simple montrant comment cela fonctionne:

from flask import request

@app.route('/upload', methods=['GET', 'POST'])
def upload_file():
    if request.method == 'POST':
        f = request.files['the_file']
        f.save('/var/www/uploads/uploaded_file.txt')
    ...

Si vous voulez savoir comment le fichier a été nommé sur le client avant qu'il ne soit téléchargé sur votre application, vous pouvez accéder à la filename attribut. Cependant, gardez à l'esprit que cette valeur peut être forgée ne jamais faire confiance à cette valeur. Si vous souhaitez utiliser le nom de fichier du client pour stocker le fichier sur le serveur, passez-le à travers le secure_filename() fonctionne que Werkzeug vous fournit:

from werkzeug.utils import secure_filename

@app.route('/upload', methods=['GET', 'POST'])
def upload_file():
    if request.method == 'POST':
        file = request.files['the_file']
        file.save(f"/var/www/uploads/{secure_filename(file.filename)}")
    ...

Pour de meilleurs exemples, voir Téléchargement de Fichiers.

Cookies¶

Pour accéder aux cookies, vous pouvez utiliser cookies attribut. Pour définir les cookies, vous pouvez utiliser set_cookie méthode des objets de réponse. Le cookies l'attribut des objets de requête est un dictionnaire avec tous les cookies que le client transmet. Si vous souhaitez utiliser sessions, n'utilisez pas les cookies directement, mais utilisez plutôt les cookies Sessions dans Flask qui ajoutent une certaine sécurité sur les cookies pour vous.

Cookies de lecture:

from flask import request

@app.route('/')
def index():
    username = request.cookies.get('username')
    # use cookies.get(key) instead of cookies[key] to not get a
    # KeyError if the cookie is missing.

Stockage des cookies:

from flask import make_response

@app.route('/')
def index():
    resp = make_response(render_template(...))
    resp.set_cookie('username', 'the username')
    return resp

Notez que les cookies sont définis sur les objets de réponse. Puisque vous normalement il suffit de retourner les chaînes des fonctions de vue Flask les convertira en objets de réponse pour vous. Si vous voulez explicitement le faire, vous pouvez utiliser le make_response() fonctionnez et modifiez-le.

Sometimes you might want to set a cookie at a point where the response object does not exist yet. This is possible by utilizing the Deferred Request Callbacks pattern.

For this also see About Responses.

Redirects and Errors¶

To redirect a user to another endpoint, use the redirect() function; to abort a request early with an error code, use the abort() function:

from flask import abort, redirect, url_for

@app.route('/')
def index():
    return redirect(url_for('login'))

@app.route('/login')
def login():
    abort(401)
    this_is_never_executed()

This is a rather pointless example because a user will be redirected from the index to a page they cannot access (401 means access denied) but it shows how that works.

By default a black and white error page is shown for each error code. If you want to customize the error page, you can use the errorhandler() decorator:

from flask import render_template

@app.errorhandler(404)
def page_not_found(error):
    return render_template('page_not_found.html'), 404

Note the 404 after the render_template() call. This tells Flask that the status code of that page should be 404 which means not found. By default 200 is assumed which translates to: all went well.

See Handling Application Errors for more details.

About Responses¶

The return value from a view function is automatically converted into a response object for you. If the return value is a string it’s converted into a response object with the string as response body, a 200 OK status code and a text/html mimetype. If the return value is a dict or list, jsonify() is called to produce a response. The logic that Flask applies to converting return values into response objects is as follows:

  1. If a response object of the correct type is returned it’s directly returned from the view.

  2. If it’s a string, a response object is created with that data and the default parameters.

  3. If it’s an iterator or generator returning strings or bytes, it is treated as a streaming response.

  4. If it’s a dict or list, a response object is created using jsonify().

  5. If a tuple is returned the items in the tuple can provide extra information. Such tuples have to be in the form (response, status), (response, headers), or (response, status, headers). The status value will override the status code and headers can be a list or dictionary of additional header values.

  6. If none of that works, Flask will assume the return value is a valid WSGI application and convert that into a response object.

If you want to get hold of the resulting response object inside the view you can use the make_response() function.

Imagine you have a view like this:

from flask import render_template

@app.errorhandler(404)
def not_found(error):
    return render_template('error.html'), 404

You just need to wrap the return expression with make_response() and get the response object to modify it, then return it:

from flask import make_response

@app.errorhandler(404)
def not_found(error):
    resp = make_response(render_template('error.html'), 404)
    resp.headers['X-Something'] = 'A value'
    return resp

APIs with JSON¶

A common response format when writing an API is JSON. It’s easy to get started writing such an API with Flask. If you return a dict or list from a view, it will be converted to a JSON response.

@app.route("/me")
def me_api():
    user = get_current_user()
    return {
        "username": user.username,
        "theme": user.theme,
        "image": url_for("user_image", filename=user.image),
    }

@app.route("/users")
def users_api():
    users = get_all_users()
    return [user.to_json() for user in users]

This is a shortcut to passing the data to the jsonify() function, which will serialize any supported JSON data type. That means that all the data in the dict or list must be JSON serializable.

For complex types such as database models, you’ll want to use a serialization library to convert the data to valid JSON types first. There are many serialization libraries and Flask API extensions maintained by the community that support more complex applications.

Sessions¶

In addition to the request object there is also a second object called session which allows you to store information specific to a user from one request to the next. This is implemented on top of cookies for you and signs the cookies cryptographically. What this means is that the user could look at the contents of your cookie but not modify it, unless they know the secret key used for signing.

In order to use sessions you have to set a secret key. Here is how sessions work:

from flask import session

# Set the secret key to some random bytes. Keep this really secret!
app.secret_key = b'_5#y2L"F4Q8z\n\xec]/'

@app.route('/')
def index():
    if 'username' in session:
        return f'Logged in as {session["username"]}'
    return 'You are not logged in'

@app.route('/login', methods=['GET', 'POST'])
def login():
    if request.method == 'POST':
        session['username'] = request.form['username']
        return redirect(url_for('index'))
    return '''
        <form method="post">
            <p><input type=text name=username>
            <p><input type=submit value=Login>
        </form>
    '''

@app.route('/logout')
def logout():
    # remove the username from the session if it's there
    session.pop('username', None)
    return redirect(url_for('index'))

How to generate good secret keys

A secret key should be as random as possible. Your operating system has ways to generate pretty random data based on a cryptographic random generator. Use the following command to quickly generate a value for Flask.secret_key (or SECRET_KEY):

$ python -c 'import secrets; print(secrets.token_hex())'
'192b9bdd22ab9ed4d12e236c78afcb9a393ec15f71bbf5dc987d54727823bcbf'

A note on cookie-based sessions: Flask will take the values you put into the session object and serialize them into a cookie. If you are finding some values do not persist across requests, cookies are indeed enabled, and you are not getting a clear error message, check the size of the cookie in your page responses compared to the size supported by web browsers.

Besides the default client-side based sessions, if you want to handle sessions on the server-side instead, there are several Flask extensions that support this.

Message Flashing¶

Good applications and user interfaces are all about feedback. If the user does not get enough feedback they will probably end up hating the application. Flask provides a really simple way to give feedback to a user with the flashing system. The flashing system basically makes it possible to record a message at the end of a request and access it on the next (and only the next) request. This is usually combined with a layout template to expose the message.

To flash a message use the flash() method, to get hold of the messages you can use get_flashed_messages() which is also available in the templates. See Message Flashing for a full example.

Logging¶

Changelog

New in version 0.3.

Sometimes you might be in a situation where you deal with data that should be correct, but actually is not. For example you may have some client-side code that sends an HTTP request to the server but it’s obviously malformed. This might be caused by a user tampering with the data, or the client code failing. Most of the time it’s okay to reply with 400 Bad Request in that situation, but sometimes that won’t do and the code has to continue working.

You may still want to log that something fishy happened. This is where loggers come in handy. As of Flask 0.3 a logger is preconfigured for you to use.

Here are some example log calls:

app.logger.debug('A value for debugging')
app.logger.warning('A warning occurred (%d apples)', 42)
app.logger.error('An error occurred')

The attached logger is a standard logging Logger, so head over to the official logging docs for more information.

See Handling Application Errors.

Hooking in WSGI Middleware¶

To add WSGI middleware to your Flask application, wrap the application’s wsgi_app attribute. For example, to apply Werkzeug’s ProxyFix middleware for running behind Nginx:

from werkzeug.middleware.proxy_fix import ProxyFix
app.wsgi_app = ProxyFix(app.wsgi_app)

Wrapping app.wsgi_app instead of app means that app still points at your Flask application, not at the middleware, so you can continue to use and configure app directly.
